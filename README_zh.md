# Doc Harness

[English README](README.md)

**基于文档的项目控制系统，服务于 AI 与人类的协作。**

Doc Harness 是一个 [Claude Code](https://claude.ai/code) skill，为你和 AI 智能体提供一套结构化的项目管理方式——就像马具（harness）驾驭马匹的力量，使其方向明确、步调可控。

它确保项目在不断推进中始终保持有序，让 AI 智能体按照你设定的原则工作，将每一个重要的结果和决策持久化为本地文档（而非让它们随着 AI 上下文的重置而消失），并使任何新到来的智能体（或你自己在休息之后）都能准确地从上次中断的地方继续。

适用于任何项目：撰写论文、开发 SaaS 功能、为客户做数据分析、写深度调查报道、开发软件库，或者任何跨越多个工作会话并产生文件的工作。

---

## 它解决什么问题？

当你和 AI 智能体在一个项目上合作数天乃至数周时：

- **AI 会丢失记忆。** 上下文窗口会被压缩或重置。昨天花两小时做的分析，今天全忘了。
- **文件越来越多却没有索引。** 几次工作之后你的项目里有30个文件，没人记得什么在哪里。
- **原则逐渐被遗忘。** 你定了重要规则（"必须在样本外数据上验证"），但随着工作深入，AI 渐渐忘了遵守。
- **交接失败。** 换了一个智能体，对项目一无所知。你花20分钟重新解释。
- **你自己也找不到方向。** 周末过后，你和 AI 都不确定上次做到哪里了。

Doc Harness 通过维护一套**自包含的文档体系**来解决这些问题——一组任何智能体都能阅读并立即理解的文件，无需任何外部记忆或额外说明。

---

## 什么是"项目"？

任何有目标、跨越多个工作会话、并产生文件的结构化工作：

- 学术论文或研究报告
- SaaS 功能或软件模块
- 为客户做的数据分析
- 深度报道或调研报告
- 系列实验
- 产品上线计划

项目就是**一个文件夹**。Doc Harness 在其中创建五个状态文档，与你自己的工作文件并存：

```
my-project/
├── CLAUDE.md                ← Doc Harness：稳定锚点
├── CURRENT_STATUS.md        ← Doc Harness：活跃脉搏
├── FILE_INDEX.md            ← Doc Harness：文件目录
├── WORKLOG.md               ← Doc Harness：永久档案
├── DOC_HARNESS_SPEC.md      ← Doc Harness：参考手册
│
├── notes/                   ← 你的工作（自由组织）
├── scripts/
├── data/
└── drafts/
```

Doc Harness 不限制你的文件夹结构——它只要求每个文件都注册到 FILE_INDEX 中。

---

## 五个文档：各自的作用

每个文档有不同的职责、不同的更新节奏、回答不同的问题。它们共同覆盖了智能体或协作者可能提出的所有问题。

### CLAUDE.md — 稳定锚点

*回答："这是什么项目？有什么规则？从哪里开始？"*

任何智能体读到的第一个文件。包含项目概述、不可违反的规则（"铁律"）、技术细节（编译命令、路径等），以及——关键的——**嵌入的操作规则**，教会智能体如何维护 Doc Harness。这意味着你不需要每次向新智能体解释这套体系——规则就在文件里。

**极少变化。** 只在阶段切换或新增重要规则时更新。它是一切的稳定基础。

### CURRENT_STATUS.md — 活跃脉搏

*回答："刚完成了什么？正在做什么？下一步是什么？"*

最动态的文档，结构像一辆**行驶中的汽车**：

| 区域 | 通俗名称 | 内容 |
|------|---------|------|
| 车辙 | **近期历史** | 最近2-3个已完成阶段的简要总结，指向 WORKLOG 获取详情。 |
| 车身 | **当前阶段详情** | 当前正在发生的一切——每个完成的步骤、创建的文件、做出的决策。详细的工作记录。 |
| 车灯 | **下一步** | 接下来做什么：立即行动 + 远期计划。 |
| 驾驶手册 | **阶段原则** | 当前阶段的工作准则——对本阶段重要但下一阶段未必适用的规则。 |

**每次工作都更新。** 当一个阶段结束时（比如"文献调研"完成，开始"数据分析"），**阶段切换**发生：

1. 车身（完整详情）被复制到 WORKLOG——数据优先保护
2. 车身被压缩为3-5行摘要，加入车辙
3. 超过3个的旧车辙被移除（它们安全地保存在 WORKLOG 中）
4. 车身清空，开始新阶段
5. 驾驶手册根据新阶段更新

这个循环让 CURRENT_STATUS 始终保持简洁（约200行），同时不丢失任何信息。

**这就是恢复能力的来源。** 一个记忆被清空的智能体读到 CURRENT_STATUS，几秒钟内就知道：最近完成了什么（车辙）、现在进行到哪里（车身）、下一步做什么（车灯）、要遵守什么原则（驾驶手册）。不需要任何额外说明。

### FILE_INDEX.md — 文件目录

*回答："项目里有哪些文件？是什么？在哪里？"*

每个文件都有一行记录，按类别组织：

```markdown
## 研究笔记
- `notes/literature_review.md` — 30篇论文的综述
- `notes/key_findings.md` — 跨论文核心发现汇总

## 脚本
- `scripts/analyze_data.py` — 价格冲击回归分析
```

**随文件增加而增长。** 规则很简单：创建文件 → 立即注册。未注册的文件对未来的智能体来说是不可见的——等于不存在。

### WORKLOG.md — 永久档案

*回答："三周前的第二阶段到底发生了什么？"*

当一个阶段结束时，其完整记录从 CURRENT_STATUS 移入这里。顶部的目录表确保即使有很多阶段也能快速定位。内容只增不改——一旦写入，永不修改或删除。

CURRENT_STATUS 只保留最近的摘要以保持简洁。WORKLOG 是完整历史的永久居所。

### DOC_HARNESS_SPEC.md — 参考手册

完整规范，包含设计原理、边缘情况处理和实战示例。日常工作中你不需要读它——CLAUDE.md 中的操作规则覆盖了95%的场景。规范是为了理解系统设计原理或处理特殊情况。

### 为什么这五个文档能协同工作

| 文档 | 更新频率 | 回答的问题 |
|------|---------|-----------|
| CLAUDE.md | 极少 | "这是什么项目？" |
| CURRENT_STATUS | 每次工作 | "现在在做什么？" |
| FILE_INDEX | 文件变化时 | "东西在哪里？" |
| WORKLOG | 阶段切换时 | "之前发生了什么？" |

每个事实只存在于**一个**文档中（单一事实来源），所以不会有矛盾。操作规则嵌入在 CLAUDE.md 中，任何智能体读到它就自动知道如何维护整个体系。

## 开始使用

### 首次安装

本仓库是一个 Claude Code 插件市场。有两种安装方式，选一种即可。

#### 方式 A——原生插件市场（推荐，Claude Code 已支持 /plugin 命令的版本）

在 Claude Code 中输入：

```
/plugin marketplace add cilidinezy-commits/doc-harness
/plugin install doc-harness-zh@doc-harness
```

若想安装英文版，第二行改为 `/plugin install doc-harness@doc-harness`——两者是独立插件，同时只装其一（它们暴露同一个 `/doc-harness` 命令）。

验证：输入 `/doc-harness`，应能看到 `init` 与 `check` 的帮助说明。

#### 方式 B——手动复制（任何环境都可用）

Skill 就是一个装着若干 Markdown 文件的目录。"手动安装"就是把这个目录复制到 Claude Code 的 skills 目录下。

**第一步——克隆仓库：**
```bash
git clone https://github.com/cilidinezy-commits/doc-harness.git
cd doc-harness
```

**第二步——选择语言版本。** `skill/` 是英文版，`skill-zh/` 是中文版。两者安装到同一路径，不能并存——选择与你项目文档语言一致的那个。

**第三步——复制到 Claude Code 的 skills 目录。**

Unix / macOS / Git Bash on Windows：
```bash
cp -r skill-zh ~/.claude/skills/doc-harness      # 中文
# 或：
cp -r skill ~/.claude/skills/doc-harness         # 英文
```

Windows PowerShell / cmd：
```powershell
xcopy skill-zh %USERPROFILE%\.claude\skills\doc-harness\ /E /I
```

**第四步——验证。** 在 Claude Code 任意目录中输入 `/doc-harness`，应能看到 `init` 与 `check` 的帮助说明。若命令未被识别，说明文件未复制到正确路径——检查 `~/.claude/skills/doc-harness/SKILL.md` 是否存在。

**查看已安装版本**：`head -3 ~/.claude/skills/doc-harness/spec.md`——应打印 `**Version**` 行（如 `v1.4`）。

### 项目级安装（可选）

若希望某个项目锁定特定 Doc Harness 版本、与用户级安装解耦，将 skill 放在 `<项目根>/.claude/skills/doc-harness/` 即可。Claude Code 优先使用项目级，其次用户级。`/doc-harness check` 解析安装路径的顺序：项目 → `~/.claude` → XDG → Windows 默认。

### 升级已有安装

**如果你用的是方式 A（插件市场）**：`/plugin marketplace update doc-harness`——Claude Code 会拉取最新并原地更新已安装的插件。

**如果你用的是方式 B（手动复制）**：
1. **拉取最新代码**：`cd doc-harness && git pull`
2. **重新复制 skill 文件夹**——命令与首次安装第三步相同，会原地覆盖；已安装 skill 目录中不保存本地状态，不会丢失任何内容。
3. **验证新版本**：`head -3 ~/.claude/skills/doc-harness/spec.md`。

**两种方式都需要：升级已有项目的 CLAUDE.md**（重要）：每个项目 `CLAUDE.md` 中嵌入的操作规则是 `init` 时的**快照**，**不会**随 skill 升级自动更新。要同步至最新版，请将 `CLAUDE.md` 中 `<!-- doc-harness-ops-start -->` 与 `<!-- doc-harness-ops-end -->` 之间的字节替换为新版 `operational_rules.md` 的内容。哨兵**之外**的内容（自定义铁律、项目特定章节等）保留不动。在项目中运行 `/doc-harness check`——§1.10 会告诉你嵌入版本是否陈旧。
5. **（若项目存在 `DOC_HARNESS_SPEC.md`）**：用新 `spec.md` 覆盖。
6. **v1.2 → v1.3 / v1.4 特殊情况**：若项目有跨项目依赖，按 `spec.md` §14.7 追溯启用 `inbox/outbox`。独立项目可跳过。

### 卸载

**如果你用的是方式 A（插件市场）**：`/plugin uninstall doc-harness`（中文版则是 `/plugin uninstall doc-harness-zh`）。

**如果你用的是方式 B（手动复制）**——全局移除 skill：
```bash
rm -rf ~/.claude/skills/doc-harness
```
Windows：`rmdir /S %USERPROFILE%\.claude\skills\doc-harness`。

**在某个项目上停用 Doc Harness**（其他项目仍可用）：删除或移走该项目的 4 个核心文件（`CLAUDE.md`、`CURRENT_STATUS.md`、`FILE_INDEX.md`、`WORKLOG.md`）——如 `mv *.md _archive/`。没有 Doc Harness 文件的项目里，skill 本身自然失效。

**降级到旧版**：`cd doc-harness && git checkout v1.3`（或其他 tag），再重新复制。已有项目中嵌入的操作规则保持当时 `init` 时的版本。

### 使用

安装后，AI 智能体就知道 Doc Harness 了。你不需要记住命令——用自然语言交流就行：

> *"帮这个项目建立一套文档管理体系。"*
> *"我想组织一下这个项目，让我们不会丢失进度。"*
> *"为这个项目创建状态文档系统。"*

智能体会理解你的需求，询问项目的名称、目标和当前状态，然后创建五个文档。

你也可以直接使用斜杠命令：`/doc-harness init`

### 新项目和进行中的项目都可以

**新项目？** 智能体会为一个全新的开始创建文档——你的目标、第一批任务、初始规则。

**项目已经进行了一段时间？** 同样好用。告诉智能体你现在在哪里：

> *"我们这个项目已经做了两周了，数据收集阶段已经完成，现在在构建分析管道。"*

智能体会创建反映当前状态的文档——不是一张白纸。现有的文件可以逐步注册到 FILE_INDEX 中。

## 安装后会发生什么

以下是为一个真实项目生成的 CLAUDE.md 的样子：

```markdown
# 市场分析工具 — 入口文档

**最后更新**: 2026-04-02
**当前阶段**: 数据管道（⏳）
**一句话状态（截至2026-04-02）**: 核心爬虫已完成，正在构建转换层

## 恢复链

### 必读（基线）
1. 本文件（CLAUDE.md）
2. CURRENT_STATUS.md

### 任务条件读
- 如要查找具体文件：读 FILE_INDEX.md
- 如要查历史阶段详情：读 WORKLOG.md
- [随工作类别出现添加项目特定条目]

### 元规则
- 自含：只指向项目内文件。不依赖 agent 侧特性。
- 活的：阶段切换时审查。

## 项目概述
构建内部使用的市场数据分析工具。爬取订单簿数据，
进行转换，生成每日报告。Python + pandas，部署在 AWS。

## 铁律
- 绝不将 API 密钥提交到 git
- 所有数据转换必须幂等
- "落笔为安"——重要信息保存到文件并注册 FILE_INDEX

## Doc Harness — 操作规则
[约100行规则，教会任何智能体如何维护这套体系]
```

以及 CURRENT_STATUS.md：

```markdown
# CURRENT_STATUS — 市场分析工具

**最后更新**: 2026-04-02
**当前阶段**: 数据管道

## 近期历史（车辙）
### 阶段1：爬虫（2026-03-20 ~ 2026-04-01）
为3个交易所构建了订单簿爬虫，处理了频率限制和断线重连。
创建了 `scripts/scraper.py` 和 `tests/test_scraper.py`。详情见 WORKLOG。

## 当前阶段详情（车身）
### 阶段目标
构建数据转换层（原始 → 清洗 → 聚合）

### 已完成步骤
#### 搭建转换框架（2026-04-02）
- 创建了 `scripts/transform.py`，采用管道架构
- 已注册 FILE_INDEX

## 下一步（车灯）
### 立即行动
1. 实现 OHLCV 聚合
2. 添加数据质量检查

## 阶段原则（驾驶手册）
- 先写测试再写转换逻辑
- 每个阶段后验证输出格式
```

任何智能体读完这两个文件就能立即接手你的工作。

## 日常使用

**工作开始** → 智能体读 CLAUDE.md（概述+规则）→ CURRENT_STATUS（当前状态）→ 继续工作。

**工作过程中** → 完成一个步骤 → 更新 CURRENT_STATUS。创建文件 → 注册 FILE_INDEX。有重要发现 → 保存到文件（"落笔为安"）。

**阶段结束** → 五步切换：车身归档到 WORKLOG（数据优先保护），压缩为车辙摘要，车身清空进入新阶段。

**工作结束** → 快速检查：状态更新了？文件注册了？有没有未保存的信息？

**上下文重置或新智能体到来** → 读 CLAUDE.md → CURRENT_STATUS → 完全了解情况。

### 你（人类用户）需要做什么

- **大多数时候：无需额外操作。** 规则在 CLAUDE.md 中，智能体会遵守。
- **偶尔：提醒一下智能体，** 如果你注意到它没有更新状态或注册文件。自然地说一句 *"记得更新项目状态"* 或 *"那个新文件注册了吗？"* 就够了。
- **觉得有点乱的时候：** 只需说 *"检查一下项目文档的状态"* 或 *"做一下文档健康检查"*。智能体知道该怎么做。（也可以直接输入 `/doc-harness check`。）

### 非线性工作

不是所有项目都按严格的顺序推进。你可能同时在做数据分析和报告撰写，或者在不同功能之间跳转。Doc Harness 可以处理这种情况：
- 在车身中用子标题追踪**所有**活跃的工作线
- "阶段切换"发生在*整体重心*转移时，不是每次切换任务都要触发
- 拿不准的时候，不要触发阶段切换——在车身内部用标题清晰分隔就好

## 健康检查

任何时候，你都可以让智能体检查项目文档的维护状况：

> *"检查一下项目文档的健康状况。"*
> *"在后台做一下状态文档检查。"*

（或直接：`/doc-harness check`）

检查做两件事：

**文件审计** — 所有文档都在吗？CURRENT_STATUS 是最新的吗？FILE_INDEX 和实际文件一致吗？发现问题会给出具体修复建议。

**原则反思** — 读取项目的铁律和阶段原则，提示智能体思考：*"我真的在遵守这些吗？"* 同时检查：*"有没有重要信息只存在于上下文中？"*

> 建议作为后台任务运行，不打断主要工作。

---

## 核心原则："落笔为安"

AI 的上下文是临时的。分析结果、设计决策、重要洞见、调试发现——如果不在文件里，上下文重置后就会消失。而且文件必须**注册到 FILE_INDEX**——未注册的文件对未来的智能体来说是不可见的。

这一条原则防止的信息丢失比其他任何功能都多。

## 常见问题

**Q: 项目已经进行了一段时间，还能用吗？**
可以——这是常见场景。告诉智能体你项目的当前状态，它会创建反映现状的文档。现有文件可以逐步注册到 FILE_INDEX。

**Q: 我的工作不是按顺序来的，经常同时做好几件事。能用吗？**
可以。在车身中用子标题追踪并行的工作线。阶段切换发生在整体重心转移时，不是每次切换任务都要触发。

**Q: 除了 Claude Code 还能用其他 AI 工具吗？**
`/doc-harness` 命令是 Claude Code 专属的。但文档体系本身是通用的——任何能读 markdown 的 AI 都可以遵循 CLAUDE.md 中的操作规则。

**Q: 这和 Claude Code 的 MEMORY.md 是什么关系？**
互补。MEMORY.md 是 Claude 的跨会话个人记忆（自动维护）。Doc Harness 是项目级文档（显式维护）。Doc Harness 设计为不依赖 MEMORY.md——任何智能体，无论有没有个人记忆，都能从文件中恢复项目状态。

**Q: 额外负担大吗？**
很小。每完成一步写一行，每创建一个文件注册一行。健康检查约2分钟，可以在后台运行。

**Q: AI 真的会遵守规则吗？**
操作规则嵌入在 CLAUDE.md 中，智能体每次工作开始都会读到。大多数时候遵守得很好。偶尔你可能需要提醒一句——"更新一下状态"或"那个文件注册了吗"。`/doc-harness check` 能发现偏离。

**Q: 可以自定义结构吗？**
可以。添加章节、定义自己的类别、调整原则。核心要求：保留四个文档、遵循阶段切换协议、注册文件。

**Q: 子项目呢？**
每个子项目可以有自己的 Doc Harness。父项目的 FILE_INDEX 链接到子项目入口。

**Q: 我的项目有 1000+ 文件，真的要一个一个注册？**
不用——这正是 FILE_INDEX 子索引和批量注册解决的问题。见 spec §4.3（子目录文件 >20 应建自己的子 FILE_INDEX.md）和 §4.4（批量注册允许一条条目记录"文件夹 + 数量 + 命名规则"，无需逐一列出）。`/doc-harness check` §1.4 会自动递归检查子索引。

**Q: 如何在某个项目上停用 Doc Harness？**
删除四个核心文件（CLAUDE.md、CURRENT_STATUS.md、FILE_INDEX.md、WORKLOG.md），或移入 `_archive/`。安装在 `~/.claude/skills/doc-harness/` 的 skill 本身不是每项目一份——没有 Doc Harness 文件的项目里它自然失效。

**Q: 跨项目通讯——收件消息长什么样？**
一个完整示例（接收方 `inbox/` 中的 `2026-04-19-from-whoami-api-deadline.md`）：

```markdown
---
from: whoami
to: lit-system-api
date: 2026-04-19
subject: API 截止日期提前到 5 月 15 日
status: unread
in-reply-to: null
priority: high
---

提醒——学术求职站 2026-05-20 上线，需要你的 citation-marker API
在 2026-05-15 前可用。契约不变（见 2026-04-10 规范），只是截止提前。

请通过发送回复消息确认收到。
```

接收方 agent 读取后将 `status: unread` → `read`，执行所需工作（此处：确认 + 计划），然后 → `actioned`。完整协议：spec 第 14 章。通过 init 启用（第一步 y/n 询问）或按 spec §14.7 在已有项目中追溯启用。

**Q: v1.2 新增了什么？**
- **恢复链现为两层结构**：精简的必读基线（2–3 个文件）加上按任务条件阅读的列表。自含——不依赖 agent 侧的 memory 或外部服务。
- **WORKLOG 归档**：`WORKLOG.md` 超过约 1000 行时，较早的阶段迁至 `WORKLOG_ARCHIVE_<YYYY-QN>.md`（按季度）。活跃 WORKLOG 保持可读，历史不丢。
- **两份可选文档**用于长期内容：`PARKING_LOT.md`（带复活前置条件的暂缓事项）和 `PHILOSOPHY.md`（项目实践催生的原则）。均为可选——有内容再创建。

**Q: v1.3 新增了什么？**
- **跨项目 inbox/outbox** 现为 Doc Harness 的可选功能（spec 第14章）。当项目需要与其他项目协调时，可启用 `inbox/` 和 `outbox/` 目录，使用带 YAML 前置信息的 Markdown 消息协议。完整规范自含于每个项目的 `DOC_HARNESS_SPEC.md` 中，无需外部文档。*（这一决定与 v1.2 "inbox/outbox 不属于 Doc Harness 范围"相反；设计理由见 `spec.md` 附录 E。）*
- `/doc-harness init` 现在会询问是否启用 inbox/outbox 协议，并自动设置文件夹、铁律块、恢复链条目和 FILE_INDEX 类别（第3.6步）。
- `/doc-harness check` 现在审计收件箱未读消息数（§1.7），并检查恢复链的两层结构健康状态（§2.5）。
- 规范中移除了**层级化的"portfolio（项目组合）"框架**。项目组内的每个项目都是自包含的对等体；父级导航文件只是一种轻量级的可选模式，不是 Doc Harness 概念。（中性术语"项目群" / §10.2 保留，用于指代这种扁平对等关系。）
- **上下文感知的更新频率**：操作规则现在指示运行环境暴露 context 使用率的智能体，将剩余 context 偏低（~<20%）视为立即触发 CURRENT_STATUS 更新及可能的阶段切换的信号。压缩等同于非自愿的 session 结束——别等那个可能永远不会到来的"有意义的步骤"。

**Q: v1.4 新增了什么？**
六轮审查驱动的全面加固：
- **Mid-transition 检测（§6.3.1）**：三值一致性检查（CLAUDE.md 阶段 / WORKLOG TOC / CURRENT_STATUS 车辙）检测被中断的阶段切换并确定性修复。接入 `/doc-harness check` §1.9。
- **异常消息处理**：§14.8 规定隔离（`inbox/_malformed/`）、分类和各类异常的严格/宽松策略。
- **陈旧消息归档触发**：§14.4 规则 3 现已可执行——当 ≥5 条 `actioned` 消息超过 30 天时，`/doc-harness check` §1.7(b) 标记归档。
- **子索引递归带剪枝**：`/doc-harness check` §1.4 正确处理嵌套 `FILE_INDEX.md`，不在父索引产生虚假幽灵。
- **版本漂移检测**：`<!-- doc-harness-ops-start/end -->` 哨兵界定嵌入的操作规则区域；check §1.10 读取版本标签并在 skill 升级后标记陈旧嵌入。重新嵌入仅替换该区域，保留用户在其下添加的自定义铁律。
- **暂停/恢复决策树（§6.4）**：三种路径——有序、紧急（mid-transition）、自动恢复（≤7 天 / 8–30 天 / >30 天 边界）。自动恢复先执行 §6.3.1。
- **收件箱文件名消歧**：§14.3 为同日冲突加 `HHMMSS` 后缀；亚秒冲突再加 `-<N>` 计数器。
- **发送前验证**：§14.3.1 要求发送方在写入前确认收件方已采用协议；送达失败记录在发送方 CURRENT_STATUS 中。
- **驾驶手册审查仪式（§6.2.2）**：对每条阶段原则应用五个问题——提升/保留/改写/移除。
- **量化 context 感知阈值**：§11.2 第 4 条具体定义"大量未保存工作"。
- **check.md 语言无关**：锚点兼容 `init` 模板产出的中英两种形式。
- **skill-creator 结构审查**：SKILL.md 描述更具触发性；spec.md 顶部新增 TOC；allowed-tools 显式包含 `Edit`；§3.1 将汽车比喻以叙事形式完整解释。
