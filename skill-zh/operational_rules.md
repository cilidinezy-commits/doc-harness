# 状态文档体系——操作规则

> 本节嵌入项目CLAUDE.md，是Agent维护状态文档的完整操作指南。
> 完整设计原理见 `STATUS_DOC_SYSTEM_SPEC.md`（项目根目录）。

---

## 首要原则："落笔为安"

Context中的信息早晚彻底丢失。重要信息必须**写入文件**并**注册到FILE_INDEX**。不写=丢失，不注册=虽存在但不可发现=等同于丢失。宁可多写一个文件，不可让信息只存在于context中。

## 四个核心文档

| 文档 | 职责 | 更新时机 |
|------|------|---------|
| **CLAUDE.md** | 项目入口。项目概述、恢复链、铁律、本操作规则 | 阶段切换时更新正文；"一句话状态"每session结束时刷新 |
| **CURRENT_STATUS.md** | 活跃状态（详见下方） | 每个session |
| **FILE_INDEX.md** | 按类别索引项目所有文件 | 创建/删除文件时 |
| **WORKLOG.md** | 完整工作历史（只增不改） | 阶段切换时从CURRENT_STATUS接收 |

## CURRENT_STATUS.md 的四个区域

按此顺序排列，每个区域用 `##` 分隔：

**§1 近期已完成（车辙）**：最近2-3个已完成阶段的**简要总结**（各3-5行）。需要详情→指向WORKLOG对应章节。新项目尚无已完成阶段时写"（项目刚启动，暂无已完成阶段）"。

**§2 当前工作（车身）**：当前阶段的**全部详细记录**——阶段目标、每个已完成步骤（做了什么、创建了什么文件、关键决策）、待解决问题。按步骤用 `####` 组织，步骤标题含日期（如 `#### 完成BibTeX验证（2026-04-01）`）。**控制在约200行以内**，超过则考虑触发阶段切换。

**§3 下一步（车灯）**：立即行动（1-3条具体步骤） + 后续规划。

**§4 当前工作原则（驾驶手册）**：本阶段适用的注意事项。随阶段生灭——阶段切换时，逐条检查：(a) 应提升为CLAUDE.md铁律？(b) 仍适用于新阶段→保留？(c) 已不适用→移除。原则在阶段内被满足时标注 `[已满足]` 但保留到阶段结束。

## FILE_INDEX.md 规则

- 按**类别**（`##`标题）组织，不按时间。类别是项目特定的，可随时新增/重组
- 每条：`路径` — 一句话说明
- 文件横跨两类→只放主类别，另一类写"→ 另见 §[主类别]"
- 某类别文件>10个 → 建议在其文件夹中创建子FILE_INDEX.md，父索引一行指向它
- 某类别文件>20个 → 必须创建子索引
- 批量创建同类文件 → 可注册文件夹+数量+命名规则，而非逐一注册

## WORKLOG.md 规则

- 头部维护**目录表**，格式：

```markdown
## 目录
| 阶段 | 时间 | 锚点 |
|------|------|------|
| 阶段2: [名称] | YYYY-MM-DD~MM-DD | [→](#阶段2-名称) |
| 阶段1: [名称] | YYYY-MM-DD~MM-DD | [→](#阶段1-名称) |
```

- 每个阶段一个`##`章节，含"阶段总结"（3-5行）+"详细记录"（从CURRENT_STATUS车身复制）
- 最新阶段在最前面（目录之后）
- 新项目的WORKLOG只有目录（空表），第一次阶段切换时才有内容
- 已写入内容**不修改不删除**。事实错误用勘误标注：`[勘误(日期): 更正内容]`

### 归档（WORKLOG 增长过长时）

**触发条件**：WORKLOG.md 超过约 1000 行（按行数判断，不按阶段数——各项目阶段密度差异极大）。

**归档流程**：
1. `WORKLOG.md` 中保留最近 3 个阶段。
2. 将更早的阶段（连同它们的目录条目）移入 `WORKLOG_ARCHIVE_<YYYY-QN>.md`（按季度粒度，如 `WORKLOG_ARCHIVE_2026-Q1.md`）。归档文件结构与 WORKLOG 相同（顶部目录 + 阶段章节）。
3. 在 `WORKLOG.md` 顶部添加指针：`> 更早的历史归档于：WORKLOG_ARCHIVE_2026-Q1.md, ...`
4. 在 FILE_INDEX 的 `## 历史归档` 类别下注册每个归档文件。

归档文件是永久记录，不得删除。

## 阶段切换——五步操作

**何时切换**：工作目标质变、用户明确宣布、暂停>3天、车身>200行。同一目标内的子任务切换（如Ch4→Ch5都是"文献补强"）不切换，在车身内用子标题分隔。拿不准→不切换。

**五步（严格按此顺序）**：

1. **保护数据**：将CURRENT_STATUS车身全部内容复制到WORKLOG顶部（目录之后），添加阶段总结。→ 此步完成后即使中断也不丢数据。
2. 在CURRENT_STATUS车辙区最顶部插入新摘要（3-5行）。标注"详细记录见WORKLOG §[阶段标题]"。
3. 车辙超过3个→删除最老的（已在WORKLOG中永久保存）。
4. 清空车身→填入新阶段目标。可在车身最前面写一行简短的上下文（如"承接阶段N的XXX成果"），但详细内容不重复（在WORKLOG中）。更新驾驶手册（逐条审查旧原则）。
5. 更新CLAUDE.md的"当前阶段"和"一句话状态（截至日期）"。更新WORKLOG目录表。

## 恢复链（Session开始）

恢复链在 CLAUDE.md 中定义，分两层：

**必读**（2–3个文件，任何继续工作的基线）：
1. CLAUDE.md（项目概述、铁律、操作规则）
2. CURRENT_STATUS.md（车辙/车身/车灯/驾驶手册）

**任务条件读**（仅当工作匹配时读）：
- 如要查找具体文件：读 FILE_INDEX.md
- 如要查历史阶段详情：读 WORKLOG.md
- [随时间累积项目特定条目]

**元规则**：
- 恢复链必须**自含**：条目只指向项目内文件（或兄弟项目的稳定路径）。不依赖 agent 侧特性（memory 系统、对话历史、外部服务）——这些对下一个 agent 可能不存在。
- 恢复链是**活的**：阶段切换时审查——新工作类别出现则增加条目，旧类别退役则删除。

**session 开始时应用**：
1. 按顺序读必读层。
2. 扫描任务条件读，只读条件匹配当前工作的条目。
3. 用户在场→确认下一步是否有变。Compact 恢复且用户不在场→直接按车灯继续。

## 工作过程中

- 每完成一个**有意义的步骤**→更新CURRENT_STATUS车身。"有意义"=如果此刻丢失context，下一个Agent需要知道这一步已完成。
- 创建新文件→**同时做两件事**：(1)注册FILE_INDEX (2)记录在CURRENT_STATUS车身。
- 重要信息→**立即**写入文件并注册（落笔为安）。
- **关注剩余 context**（如运行环境暴露该信息）。压缩等同于非自愿的 session 结束。剩余 context 降低（~<20%）时，在下一次工具调用**之前**更新 CURRENT_STATUS；如果车身积累了大量未保存的工作，立即考虑阶段切换。别等下一个"有意义的步骤"——它可能来不及了。

## Session结束检查清单

- [ ] CURRENT_STATUS车身反映了本session全部工作？
- [ ] 车灯的"下一步"是准确的？
- [ ] CURRENT_STATUS的"最后更新"日期已刷新？
- [ ] CLAUDE.md的"一句话状态（截至日期）"已刷新？
- [ ] 本session创建的文件都注册了FILE_INDEX？
- [ ] 有没有重要信息只存在于context中？→落笔为安！
- [ ] 本session 发生了阶段切换？→ WORKLOG 在 1000 行内？（超则触发归档。）

> 用户可随时要求执行 `/doc-harness check` 进行全面的状态文档健康检查与工作原则反思。建议以后台agent方式运行，避免打断当前工作。

## 项目暂停

执行阶段切换五步→在车身写"项目暂停于YYYY-MM-DD。原因：..."→CLAUDE.md状态改为⏸️。恢复时注意：车灯的"下一步"可能已过期，需与用户确认。

## 单一事实来源

每个事实只在一个文档中维护。当前状态/数字→CURRENT_STATUS。文件清单→FILE_INDEX。历史详情→WORKLOG。项目概述/铁律→CLAUDE.md。CLAUDE.md允许一个"一句话状态（截至日期）"作为唯一冗余，矛盾时以CURRENT_STATUS为准。

## 铁律的管理

铁律存放在CLAUDE.md中，与项目同寿。可以在阶段切换时新增铁律（从驾驶手册中提升）。铁律一般不删除，如果某条铁律的前提条件不再适用，标注"[已不适用: 原因]"但保留原文。

## 文档生命周期

被替代的文档在**首行**添加：`⚠️ SUPERSEDED BY [新文档路径]`。然后选择：
- **保留作为参考**（最常见）：在FILE_INDEX中移至 `## 已替代` 类别。注明保留原因。
- **不再有用**：移入 `_archive/`，从FILE_INDEX移除。

## 可选长期文档

除核心四文档外，提供两份**可选**文档。仅当项目真的有相应内容时才创建；放入恢复链的任务条件读层，不进入必读。

- **`PARKING_LOT.md`** — 暂缓事项：想做但现在做不了的具体事。每条记录：做什么、为何推迟、复活的前置条件、何时复查。不得悄悄删除；复活或放弃时标注决定。适用于那些本会被新工作覆盖但不应遗忘的 CURRENT_STATUS 条目。

- **`PHILOSOPHY.md`** — 来自实践的原则：由本项目具体工作催生的、可推广的教训。每条记录：原则陈述、催生它的实践、适用范围、首次记录日期。原则可复制到兄弟项目的 PHILOSOPHY.md 中；本文件保留为*本项目*的诞生记录。适用于那些洞察可能超越当下情境的原则。

完整格式、生命周期与设计原理见 `DOC_HARNESS_SPEC.md` 第13章。
