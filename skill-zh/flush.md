# Doc Harness — 冲刷（Flush）

紧急保存：在上下文压缩或会话结束前，系统性地将上下文中的重要信息提取到文档中并注册。

**目的**：当上下文即将丢失（压缩、重置或用户主动结束会话）时，`flush` 确保目前**只存在于上下文中**的信息被写入文件、注册到 FILE_INDEX，并对未来智能体可发现。这是"落笔为安"失败模式的终极防线。

**核心保证**：成功冲刷后，一个全新智能体阅读项目的恢复链，应能恢复到**仿佛上下文从未被压缩**的状态——未必立即知道每个细节，但知道*存在哪些信息*以及*到哪里去找*。

> ⚠️ **常见失败模式——请先阅读**
>
> 智能体频繁在执行 `flush` 时只运行 `sync`（更新现有文档），然后**悄悄跳过阶段 B 和阶段 C**。这会产生看起来像成功冲刷的输出，但完全不包含上下文提取——没有新建文件、没有上下文信息盘点、没有阶段 B 报告。
>
> **如果阶段 B 和阶段 C 被跳过，冲刷即失败。它与 sync 毫无区别。**
>
> 冲刷的区分特征是**强制的上下文盘点与提取**。一个没有执行阶段 B 的智能体，无论输出标题声称什么，都没有执行冲刷。

---

## 何时使用

- 用户说"保存一切"、"冲刷上下文"、"为压缩做准备"或"确保不丢东西"
- 上下文使用率很高，用户希望确保零信息丢失
- 已知上下文即将重置之前（例如切换到不同对话、关闭 IDE）
- 长时间高密度工作会话之后，大量决策和分析累积在上下文中
- 当用户担心太多重要状态"在智能体脑子里"而非在磁盘上

---

## 与 `sync` 的关系

`flush` **包含 `sync` 的所有工作**作为第一阶段。差异在于强制的**上下文盘点**（第二阶段）和**提取**（第三阶段）。

若项目没有漂移（所有文件已注册、日期最新、车身未超限），`sync` 是无操作。`flush` **仍会执行上下文盘点和提取**——这是它存在的全部理由。

> **经验法则**：如果冲刷输出不包含阶段 B（盘点报告）和阶段 C（新建/追加文件），那么智能体执行的是 sync，不是 flush。纠正并重新执行。

## 两种模式

### `interactive`（默认）

在每次重要提取之前询问用户：
- "我在上下文中发现 [X 分析]。保存为 [path/xxx.md] 吗？（是/否/修改路径）"
- "我发现 [Y 决策理由]。追加到车身还是创建 [design/y.md]？"
- "我发现 [Z 教训]。保存到 PHILOSOPHY.md 吗？"

这是默认模式，因为上下文提取涉及"什么重要、该放哪里"的判断。用户应在文档创建前进行审核。

适用于：
- 用户说"flush"而没有附加条件
- 上下文包含不应被持久化的敏感或草稿信息
- 项目有严格的文件夹约定，启发式可能违反

### `auto`（`--auto` 或 `-a`）

执行完整流程，无需询问用户，使用启发式规则对上下文信息进行分类和路由。

**"重要"信息启发式**（满足任一即可）：
- 用户明确强调过（"这很重要"、"记住这个"）
- 代表决策、理由或权衡（不是转瞬即逝的想法）
- 产生它需要显著 effort（>5 分钟分析、多步推理或外部数据查询）
- 丢失它会导致下一个智能体困惑或重复工作

**目标路径启发式**：
- 若存在清晰类别，使用现有项目文件夹约定
- 若无约定，按需创建标准文件夹：`notes/`、`design/`、`lessons/`、`data/`、`tmp/`
- 绝不覆盖现有文件；总是优先"创建新文件"或"追加到现有文件"

适用于：
- 用户明确说"flush auto"或添加了 `--auto`
- 优先选择速度和自动化而非人工审核

## 执行步骤

### 阶段 A：执行同步

以相同模式（自动或交互）执行完整的 `sync` 流程（见 `sync.md`）。这确保文档基础在添加新提取之前是稳固的。

若 sync 触发阶段切换或 WORKLOG 归档，在进入阶段 B 之前完整完成。

**阶段 A 完成门控**：进入阶段 B 之前，必须显式声明：

```
阶段 A（同步）完成。
[同步动作摘要]
现在进入阶段 B：上下文盘点——强制、不可跳过。
```

---

### 阶段 B：上下文盘点（强制——不可跳过）

> 🚫 **本阶段不可跳过。** 跳过阶段 B 意味着冲刷失败。

智能体对自身当前上下文进行自我扫描。将每个非瞬态条目分类到下方类别之一。

**跳过规则**：若某条目已在 CURRENT_STATUS 车身中被引用**或**已注册在 FILE_INDEX 中，标记为 `已持久化` 并跳过——它已经是可发现的。

| 上下文信息类型 | 目标文档 | 示例 |
|----------------|---------|------|
| 分析结果 | `notes/` 或 `findings/` | "回归显示 R² = 0.87，p<0.01 显著" |
| 数据发现 | `notes/findings/` + 注册 | "数据集在 X 列有 12% 缺失值" |
| 设计决策 | `design/` 或车身 | "为支持并发写入选择 PostgreSQL 而非 SQLite" |
| 架构选择 | `design/` 或车身 | "微服务边界划在支付与库存之间" |
| 错误教训 | `PHILOSOPHY.md` 或 `lessons/` | "绝不信任 CSV 编码头——始终嗅探" |
| 错误模式 | `PHILOSOPHY.md` | "v0.3 之前反复忘记验证空输入" |
| 新规则/约束 | CLAUDE.md 铁律或驾驶手册 | "所有 API 响应必须包含 request_id" |
| 用户口头要求 | CURRENT_STATUS 车身 | "用户希望下个版本支持暗黑模式" |
| 中间数据 | `data/` 或 `tmp/` | "最终聚合前的清洗数据集" |
| 调试发现 | `notes/debug/` 或车身 | "线程池竞争条件通过添加屏障修复" |
| 跨项目依赖 | `outbox/` 消息（如已启用 inbox/outbox） | "WhoAMI 需要我们在周五前提供模型准确率" |

**排除项**（绝不提取）：
- 未收敛的 fleeting thoughts 或头脑风暴
- 已知错误或已被取代的信息
- 瞬态工具输出（例如 `ls` 列表、临时错误消息）
- 用户明确要求不保存的信息

**阶段 B 完成检查清单**（进入下一阶段前必须产出）：

```
阶段 B（上下文盘点）完成。
- 扫描条目总数: [N]
- 已持久化条目（已在文件中）: [M]
- 分类为待提取: [K]
- 排除条目（瞬态/错误/用户拒绝）: [P]
```

若 `K > 0` → 进入阶段 C。
若 `K = 0` → 产出下方的**空扫描报告**，然后进入阶段 D。

#### 空扫描报告（K = 0 时强制产出）

即使没有发现可提取条目，智能体也必须记录扫描了什么以及为何没有条目符合条件：

```markdown
#### 上下文盘点——空扫描（YYYY-MM-DD HH:MM）
- 扫描范围: [例如"本次会话的全部分析结果、设计决策、调试发现和用户要求"]
- 已持久化条目: [N]（已在车身或 FILE_INDEX 中——列于下方）
- 排除条目: [P]（瞬态 / 已被取代 / 用户拒绝——列于下方）
- 判断依据: [为何其余上下文被判定为不可提取]
```

> **为何重要**：带记录的空扫描证明阶段 B 已被执行。缺失的阶段 B 章节证明它被跳过了。

---

### 阶段 C：写入并注册（若 K > 0 则强制——不可跳过）

对每个未标记为 `已持久化` 的盘点条目：

1. **确定目标路径**
   - 自动：应用上述启发式
   - 交互：向用户提议；接受、拒绝或编辑路径

2. **写入内容**
   - 若目标文件存在 → 以日期标题追加（`## 于 YYYY-MM-DD 添加`）
   - 若目标文件不存在 → 创建，带前置信息风格标题：
     ```markdown
     # [标题]
     
     > 于 YYYY-MM-DD 从上下文中通过 `/doc-harness flush` 提取
     
     [内容]
     ```

3. **注册到 FILE_INDEX**
   - 在适当类别下添加文件路径
   - 若类别不存在，创建它
   - 若批量注册更合适（多个同类文件），注册文件夹 + 数量

4. **在 CURRENT_STATUS 车身中记录**
   - 添加一行条目：`- (YYYY-MM-DD) 冲刷: 创建/更新 [路径] — [一句话描述]`
   - 这确保下一个智能体即使不读新文件也能看到提取事件

**阶段 C 完成检查清单**：

```
阶段 C（写入与注册）完成。
- 新建文件: [路径列表]
- 追加文件: [路径列表]
- FILE_INDEX 类别已更新: [列表]
- 车身条目已添加: [数量]
```

---

### 阶段 D：验证

智能体执行模拟的全新到达：

```
模拟: "我是一个零上下文的全新智能体。我读了 CLAUDE.md → CURRENT_STATUS。
   从 CURRENT_STATUS 车身中，我是否看到了我刚才创建的所有文件的引用？
   从 FILE_INDEX 中，我能按类别定位每个新文件吗？
   从恢复链中，如果我需要它们的内容，我会知道去读这些文件吗？"
```

- **若存在任何缺口** → 补充：在车身中添加缺失引用，确保 FILE_INDEX 类别清晰，若新文档类别需要则添加恢复链任务条件读条目。
- **若全部覆盖** → 通过。

**验证检查清单**：
```
阶段 D（验证）完成。
- 全新到达模拟: [通过 / 发现缺口并已修复]
- 已修复缺口: [列表，或"无"]
```

---

### 阶段 E：最终冲刷标记

追加到 CURRENT_STATUS 车身：

```markdown
#### 上下文冲刷于 YYYY-MM-DD HH:MM — N 项提取到文件
- 同步动作: [阶段切换? 是/否] [归档? 是/否]
- 新建文件: [列表]
- 追加文件: [列表]
- 空扫描: [是/否——若是，引用上方的空扫描报告]
```

此标记作为冲刷事件的持久记录，使未来智能体知道曾进行过系统性保存。

## 输出格式

```
═══════════════════════════════════════
  Doc Harness — 上下文冲刷
  项目: [名称]
  日期: YYYY-MM-DD HH:MM
  模式: auto / interactive
═══════════════════════════════════════

── 阶段 A: 同步 ──
[与 sync.md 相同的输出]

── 阶段 B: 上下文盘点 ──
[强制——即使为空也必须出现本节]
扫描上下文。发现 N 个条目：
  [1] [类型] — [一句话摘要] → [目标路径] / 已持久化 / 已排除
  [2] [类型] — [一句话摘要] → [目标路径] / 已持久化 / 已排除
  ...
  [M] 个条目已持久化（跳过）
  [P] 个条目已排除（原因列于下方）

若 K = 0:
  空扫描报告:
  - 扫描范围: ...
  - 判断依据: ...

── 阶段 C: 写入与注册 ──
[若 K > 0: 列出新创建/追加的文件和注册信息]
[若 K = 0: "无条目需要提取。阶段 C 按空扫描报告跳过。"]

── 阶段 D: 验证 ──
模拟全新到达: [通过 / 发现缺口并已修复]

── 阶段 E: 冲刷标记 ──
已记录到 CURRENT_STATUS 车身。

═══════════════════════════════════════
```

> **如果上述输出不包含阶段 B 章节**，则智能体未能执行冲刷。视为未完成操作并要求重新执行。

## 自动模式启发式细节

### "重要"阈值

信息**重要**，若**任一**为真：
1. 用户明确点名（"重要"、"记住这个"、"记下来"）
2. 它是决策或理由（选择 X 而非 Y、为何拒绝 Z）
3. 产生它需要显著 effort（>5 分钟分析、多步推理、外部查询）
4. 丢失它会导致未来智能体重做工作或做出不一致选择
5. 它改变了先前确立的项目事实或约束

信息**不重要**，若**全部**为真：
1. 它是瞬态观察（"文件大小 4KB"）
2. 不影响未来工作
3. 可轻易复现
4. 用户没有特别指出

### 目标路径选择

优先顺序：
1. 现有项目约定（若已存在 `design/`，就使用它）
2. 按类型标准文件夹名：
   - 分析 → `notes/` 或 `analysis/`
   - 设计 → `design/`
   - 教训 → `lessons/` 或 `PHILOSOPHY.md`
   - 数据 → `data/`
   - 调试 → `notes/debug/`
3. 若需要创建文件夹且不存在，优先使用 `notes/` 作为通用兜底
