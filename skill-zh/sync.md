# Doc Harness — 同步（Sync）

将项目状态文档与现实同步。修复漂移、刷新陈旧字段、注册缺失文件，并可选择触发阶段切换或 WORKLOG 归档。

**目的**：当智能体在工作期间未能自动更新状态文档，或用户感到"已经有一段时间了——文档应该跟上进度"时，`sync` 主动修复差距。与 `check`（仅诊断）不同，`sync` 会修改文件。

---

## 何时使用

- 用户说"更新一下状态文档"、"同步项目状态"或"让文档跟上进度"
- 智能体在某次会话中未能维护文档（创建了文件但未注册、日期陈旧、车身漂移）
- 长时间暂停（>3 天）之前，确保文档反映当前现实
- 在智能体不知情的情况下进行了批量文件操作之后（例如用户手动复制了 50 个文件）
- 当 `check` 报告了用户希望立即修复的可操作漂移时

---

## 两种模式

### `interactive`（默认）

在超出常规簿记的操作之前询问用户：
- 执行阶段切换之前（车身 ≥200 行）
- WORKLOG 归档之前（≥1000 行）
- 从上下文中提取原则/教训并创建新文档之前
- 将驾驶手册原则提升为铁律之前

这是默认模式，因为阶段切换和归档是不可逆或重大的操作。除非用户明确选择自动化，否则应始终先询问。

适用于：
- 用户说"sync"而没有附加条件
- 这是项目首次 sync，且存在显著漂移
- 项目处于敏感状态，阶段切换应由人工把关

### `auto`（`--auto` 或 `-a`）

执行所有适用的修复，无需询问用户。适用于：
- 用户明确说"sync auto"或添加了 `--auto`
- 作为后台任务调用
- 优先选择速度和自动化而非人工审核

**安全性**：自动模式严格遵守 spec §6.2 的所有触发条件。它会不经确认就执行阶段切换和归档。

---

## 执行步骤

### 第一步：漂移扫描（只读）

执行针对性扫描，聚焦可操作漂移项：

| 检查项 | 方法 | 行动阈值 |
|--------|------|---------|
| 未注册文件 | 将文件系统（排除 `_archive/`、`.git/`、`node_modules/`、`dist/`、`inbox/`、`outbox/`）与 FILE_INDEX 条目对比（递归含子索引剪枝，与 `check` §1.4 相同算法） | 磁盘上存在但未在索引中列出 |
| CURRENT_STATUS 时效性 | 读取前 5 行，提取"最后更新"日期 | 不是今天 |
| CLAUDE.md 一句话状态 | 读取前 5 行，提取"一句话状态（截至日期）" | 日期与 CURRENT_STATUS 不一致 |
| 车身长度 | 统计车身标题与下一个 `##` 之间的行数 | ≥200 行 |
| WORKLOG 长度 | `wc -l` | ≥1000 行 |
| 收件箱/发件箱 | 与 `check` §1.7 相同的四个子检查（如已启用） | 任何非 ✅ 结果 |

**Token 效率**：扫描期间不读任何叶子文档内容。仅用 Grep、行计数和 `find`。

### 第二步：执行自动修复

应用所有明确安全的修复：

1. **注册未注册文件**
   - 按文件扩展名和父文件夹名启发式确定类别
   - 若无明确类别，创建或使用 `## 未分类`
   - 若注册后某类别超过 20 个文件，标记为需要创建子索引（不自动创建子索引——那需要人工判断类别边界）
   - 在 CURRENT_STATUS 车身中记录：`#### 同步注册 N 个文件（YYYY-MM-DD）`及摘要列表

2. **刷新陈旧日期**
   - 将 CURRENT_STATUS "最后更新"更新为今天
   - 将 CLAUDE.md "一句话状态（截至[日期]）"更新为今天（保留文本，仅更新日期）
   - 将 FILE_INDEX "最后更新"更新为今天

3. **收件箱/发件箱整理**（如已启用）
   - 执行与 `check` §1.7 标记相同的补救动作（陈旧 actioned 消息归档、异常审查等）

### 第三步：阶段切换检查

若车身 ≥200 行：

- **交互模式**（默认） → 询问："车身为 N 行（阈值：200）。现在执行阶段切换吗？（是/否）" 若否 → 在车身中记录：`#### 用户于 YYYY-MM-DD 推迟阶段切换（车身 N 行）`。
- **自动模式** → 立即执行五步阶段切换（spec §6.2）。数据优先保护：第一步（车身复制到 WORKLOG）在其他任何写入之前执行。

若车身 <200 行，但其中的步骤不再服务于同一阶段目标（`check` §2.4 的阶段一致性失败）：

- **自动模式** → 仍执行阶段切换。一致性失败是比行数更强的信号。
- **交互模式** → 询问："车身中的步骤似乎服务于不同目标。执行阶段切换吗？（是/否）"

### 第四步：WORKLOG 归档检查

若 WORKLOG ≥1000 行：

- **交互模式**（默认） → 询问："WORKLOG 为 N 行（阈值：1000）。现在归档较早阶段吗？（是/否）"
- **自动模式** → 按 spec §5.5 执行归档（保留最近 3 个阶段，较早的迁入 `WORKLOG_ARCHIVE_YYYY-MM-DD.md`，注册，在车身中记录，原子 git 提交）。

### 第五步：上下文原则提取（仅限交互模式）

若模式为交互：

1. 扫描当前会话的上下文，找出已浮现但尚未记录的原则、教训或规则。
2. 向用户呈现简洁摘要："在本次会话/工作阶段中，我观察到：[列表]。其中是否有应记入 PHILOSOPHY.md 或提升为 CLAUDE.md 铁律的？"
3. 若用户确认任何条目 → 创建/追加文档，注册到 FILE_INDEX，在车身中记录。

在**交互模式**（默认）下，扫描并呈现给用户确认。
在**自动模式**下，完全跳过此步骤。自动模式不尝试从上下文中综合原则——那需要人工判断哪些洞察具有普适性。

## 输出格式

```
═══════════════════════════════════════
  Doc Harness — 状态同步
  项目: [名称]
  日期: YYYY-MM-DD
  模式: auto / interactive
═══════════════════════════════════════

── 扫描结果 ──
[FILE_INDEX]    发现 N 个未注册文件
[CURRENT_STATUS] 最后更新: YYYY-MM-DD（陈旧/最新）
[CLAUDE.md]     一句话状态日期: YYYY-MM-DD（陈旧/最新）
[车身]          N 行（低于阈值/达到阈值/超出阈值）
[WORKLOG]       N 行（低于阈值/达到阈值/超出阈值）
[收件箱/发件箱]  [如已启用: 子检查摘要]

── 已执行动作 ──
- 注册 N 个文件: [类别列表]
- 刷新日期: CURRENT_STATUS、CLAUDE.md、FILE_INDEX
- 阶段切换: [不需要 / 已执行 / 用户推迟]
- WORKLOG 归档: [不需要 / 已执行 / 用户推迟]
- 记录原则: [无 / 列表]

── 待处理项（如有） ──
[交互模式下用户尚未决定的项，或自动模式下的备注]
═══════════════════════════════════════
```

## 与其他命令的关系

| | `check` | `sync` | `flush` |
|--|---------|--------|---------|
| **修改文件？** | 否 | 是 | 是 |
| **范围** | 全面审计 + 原则反思 | 漂移修复 + 维护 | sync 的一切 + 上下文提取 |
| **触发时机** | 定期 / 用户请求 / 感觉漂移 | "文档落后了" | "上下文即将压缩" |
| **阶段切换** | 报告阈值 | 可执行（自动或询问） | 可执行（自动或询问） |
| **上下文提取** | 问"有没有未保存的？"（不行动） | 交互模式才问原则 | 强制盘点 + 写入 |

---

## 语言无关性说明

所有锚点兼容 `init.md` 模板产出的**中英两种形式**——与 `check.md` 相同的双锚点逻辑。
