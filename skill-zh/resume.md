# Doc Harness — 恢复（Resume）

结构化状态恢复：当上下文为空或用户明确希望继续工作时，系统性地阅读项目状态文档、验证理解、并在继续工作前产出恢复报告。

**目的**：Doc Harness 的恢复链（在 CLAUDE.md 中定义）和 Session Start 协议（spec.md §11.1）描述了智能体到达时*应该*做什么，但它们是隐式的——智能体可能悄悄跳过，且没有证据证明理解确实被恢复了。`resume` 使状态恢复**显式、可验证、且可由用户触发**。

**核心保证**：成功恢复后，智能体能够清晰表达：
- 它是谁（项目身份）
- 项目现在正在做什么（阶段目标、活跃工作）
- 接下来应该发生什么（车灯、障碍）
- 继续工作是否安全，还是需要用户确认

> **与 Session Start 的关系**：`resume` **规范化并扩展**了 spec.md §11.1。它并不取代 Session Start。执行 `resume` 的智能体遵循恢复链，然后增加恢复报告综合与理解验证阶段。若智能体到达项目且上下文为空，它应在进行任何其他工作之前先执行 `/doc-harness resume`。

---

## 何时使用

- 用户说"继续这个项目"、"恢复工作"、"上次做到哪里了"、"回到 [项目名]"、"从停下的地方继续"
- 用户在已有 Doc Harness 文档的项目中说"当前状态是什么？"
- 智能体检测到**上下文为空** + 4 个核心文件齐全 → 在执行任何其他动作前自动触发 resume
- Compact recovery 之后（上下文被压缩，用户不在场）——在按车灯行动之前的强制第一步

---

## 两种模式

### `interactive`（默认）

智能体产出恢复报告、回答验证问题，并**将结果呈现给用户**确认或纠正后再继续。

适用于：
- 用户明确说"继续"或"恢复"
- 用户在场，可以验证智能体的理解
- 项目状态复杂或模糊，受益于人工审核

### `auto`（`--auto` 或 `-a`）

智能体执行完整流程，无需用户交互，将恢复报告写入 CURRENT_STATUS 车身，并根据新鲜度规则做出自主恢复决策。

适用于：
- Compact recovery（用户不在场，上下文重置）
- 用户明确说"resume auto"或"直接继续"
- 智能体作为后台进程运行，需要自我定向

**自动模式恢复决策树**：
- CURRENT_STATUS 今日更新、车灯 ≤7 天陈旧、无边缘条件 → 写入 `#### 于 YYYY-MM-DD 自动恢复` 到车身，按车灯继续
- 车灯 8–30 天陈旧 或 任何边缘条件被标记 → 写入 `#### 于 YYYY-MM-DD 尝试自动恢复——等待用户确认` 到车身，**不要**悄悄继续
- 车灯 >30 天 或 项目明确暂停 → 绝不自动恢复；始终等待用户明确指示

## 执行步骤

### 阶段 A：执行恢复链

严格按照 CLAUDE.md 和 spec.md §11.1 中定义的项目恢复链执行。不要跳过步骤。不要读取条件不匹配的任务条件读条目。

**步骤 0 — 身份锚定**（阅读任何文件之前）：
阅读 CLAUDE.md 顶部的 AGENT IDENTITY LOCK。确认：
"我是 [PROJECT_NAME] 的代理。我的职责是 [锁中描述的角色]。"

**步骤 1 — 必读层**：
按顺序阅读（通常 2–3 个文件）：
1. `CLAUDE.md` — 项目概述、铁律、恢复链、操作规则
2. `CURRENT_STATUS.md` — 车辙、车身、车灯、驾驶手册

**步骤 2 — 任务条件读层**：
扫描恢复链任务条件读条目。**只**阅读条件与当前情况匹配的条目：
- 若 `inbox/` 有未读消息 → 按第十四章协议阅读
- 若调查历史 → 阅读 `WORKLOG.md`
- 若查找文件 → 阅读 `FILE_INDEX.md`
- 任何项目特定条目（例如，做重大架构决策时读 PHILOSOPHY.md）

**步骤 3 — 边缘条件扫描**（进入报告前）：
按顺序执行以下检查：

1. **Mid-transition 检测**（spec.md §6.3.1）：
   - 检查 CLAUDE.md 阶段 / WORKLOG 目录表 / CURRENT_STATUS 车辙三者一致性
   - 若不一致 → **先修复**（按 §6.3.1 逐步失败表），然后继续恢复
   - 若执行了任何修复，在车身中记录

2. **暂停检查**：
   - CLAUDE.md 状态是否为 ⏸️（暂停）？
   - 若是 → 遵循 spec.md §6.4 恢复规则（确认车灯新鲜度、检查时间敏感上下文）
   - 自动模式：应用 ≤7天 / 8–30天 / >30天 决策树

3. **收件箱未读检查**：
   - `inbox/` 中是否有 `status: unread` 的消息？
   - 若有 → 标记为优先动作；不要悄悄忽略

4. **新鲜度检查**：
   - CURRENT_STATUS "最后更新"日期——今天？陈旧（>3天）？
   - 车灯条目——是否有明确日期已过的？

**阶段 A 完成检查清单**：
```
阶段 A（恢复链）完成。
- 必读文件: [列表]
- 任务条件读文件: [列表，或"无匹配"]
- 边缘条件发现: [列表，或"无"]
- 已执行修复: [列表，或"无"]
```

---

### 阶段 B：产出恢复报告（强制——不可跳过）

> **本阶段是 resume 的区分特征。** 阅读了文件但没有产出恢复报告的智能体，没有执行 resume——它只执行了 Session Start。

将阶段 A 收集的所有信息综合为以下 7 节报告。每节必须包含**智能体生成的综合**，而非复制粘贴的标题。报告证明理解。

#### 第 1 节：身份确认

```
项目: [CLAUDE.md / AGENT IDENTITY LOCK 中的名称]
我的职责: [AGENT IDENTITY LOCK 中的角色]
当前阶段: [CLAUDE.md / CURRENT_STATUS 中的阶段名称]
阶段状态: [⏳ 活跃 / ⏸️ 暂停 / 其他]
```

#### 第 2 节：当前阶段与目标

```
阶段目标: [1-2 句话总结车身的当前阶段目标]
开始时间: [如有日期]
关键约束: [任何与当前工作相关的铁律或驾驶手册原则]
```

#### 第 3 节：活跃工作摘要

```
最近完成的步骤（车身顶部 3–5 项）：
  1. [步骤标题 + 日期 + 一句话做了什么]
  2. ...

当前开放工作：
  - [进行中或未解决的内容]
```

#### 第 4 节：下一步（车灯）

```
立即行动（按车灯排名 #1、#2、#3）：
  #1: [行动 + 任何障碍]
  #2: ...

新鲜度评估：
  - 车灯最后更新: [日期]
  - 已过期项目: [列表，或"无"]
  - 车灯仍有效的置信度: [高 / 中 / 低 + 原因]
```

#### 第 5 节：未读信号

```
收件箱未读: [数量 + 主题，或"无"]
发件箱待处理（如有）: [数量 + 等待回复的收件方，或"无"]
其他需要行动的信号: [例如 check.md 发现、sync 检测到的漂移]
```

#### 第 6 节：边缘条件

```
Mid-transition  detected: [是/否——若是，修复状态]
项目已暂停: [是/否——若是，暂停日期 + 原因]
状态陈旧: [是/否——CURRENT_STATUS 最后更新 >3 天？]
关键障碍: [车身中阻止工作的任何未解决问题]
```

#### 第 7 节：智能体就绪自评

```
我已准备好继续: [是 / 否 / 有条件]
若有条件: [工作恢复前需要发生什么]
建议行动: [按车灯继续 / 等待用户 / 先运行 check|sync|flush]
```

---

### 阶段 C：理解验证（强制——不可跳过）

**用自己的话**回答以下问题，仅使用阶段 A 收集的信息。不要查找答案——这测试的是综合，而非搜索。

**问题 1 — 阶段目标**：当前阶段试图实现什么，用一句话概括？
- 通过：对阶段目标的具体、准确总结
- 失败：模糊、错误，或"我需要重新阅读文档"

**问题 2 — 下一步**：#1 下一步是什么，有什么（如有）阻碍它？
- 通过：具体行动 + 已识别的障碍或"无障碍"
- 失败：笼统行动（"继续工作"）或遗漏障碍

**问题 3 — 最后一步**：本次会话之前完成的最后一个有意义步骤是什么？
- 通过：准确识别车身中最新的条目
- 失败：错误的步骤或"不确定"

**问题 4 — 未读信号**：在正常开展工作前，是否有任何未读收件箱消息或其他信号需要处理？
- 通过：正确的数量 + 主题，或正确识别"无"
- 失败：遗漏未读消息或错误声称需要行动

**问题 5 — 安全检查**：不经过用户确认，继续执行 #1 下一步是否安全？
- 通过：正确应用自动恢复规则（≤7天新鲜 + 无边缘条件 + 未暂停 = 安全；否则 = 等待）
- 失败：条件不安全时自动继续，或条件安全时不必要地等待

**阶段 C 完成检查清单**：
```
阶段 C（理解验证）完成。
- Q1（阶段目标）: [通过 / 失败]
- Q2（下一步）: [通过 / 失败]
- Q3（最后一步）: [通过 / 失败]
- Q4（未读信号）: [通过 / 失败]
- Q5（安全检查）: [通过 / 失败]
- 总体: [全部通过 / N 处失败——已采取行动]
```

> **若任何问题失败**：重新阅读相关文档章节并重试一次。若再次失败，在恢复报告（第 7 节）中标记理解不完整，需要用户澄清。

---

### 阶段 D：恢复决策

**交互模式**：
1. 向用户呈现完整恢复报告（第 1–7 节）
2. 总结验证结果（阶段 C）
3. 询问："这与您的理解一致吗？我应该继续执行 #1 下一步，还是有变化？"
4. 将用户回复记录到 CURRENT_STATUS 车身：
   ```markdown
   #### 于 YYYY-MM-DD 恢复——用户已确认
   - 恢复报告已验证: [是 / 有修正]
   - 已记录修正: [列表，或"无"]
   - 继续执行: [#1 下一步，或用户指定的替代方案]
   ```

**自动模式**：
1. 将恢复报告（精简版）追加到 CURRENT_STATUS 车身
2. 应用自动恢复决策树（见上方"两种模式"）
3. 若继续：写入 `#### 于 YYYY-MM-DD 自动恢复` 并按车灯继续
4. 若等待：写入 `#### 于 YYYY-MM-DD 尝试自动恢复——等待用户确认` 并停止

## 与其他命令的关系

| 情况 | 命令 |
|------|------|
| 完全没有 Doc Harness 文件 | `init` |
| 文档齐全，上下文为空，需要恢复状态 | `resume` |
| 文档齐全，上下文正常，想审计健康 | `check` |
| 文档齐全，上下文正常，检测到漂移 | `sync` |
| 上下文即将压缩 | `flush` |
| 需要查找特定信息 | `recall` |

**resume vs. check**：`check` 审计文档健康（文件是否正确？）。`resume` 恢复智能体对项目状态的理解（智能体知道该做什么吗？）。一个项目可以拥有健康的文档，但智能体的上下文为空——resume 修复后者。

**resume vs. Session Start**：`resume` **就是** Session Start 加上可验证的输出。若智能体执行了 `resume`，它已履行了 Session Start 义务。若智能体手动执行了 Session Start，它没有产出恢复报告或验证——它应该执行 `resume` 来完成该过程。

所有锚点和标题匹配遵循与 `check.md` 和 `sync.md` 相同的双语逻辑。
