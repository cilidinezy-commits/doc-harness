# Doc Harness — 初始化

为新项目创建五个 Doc Harness 文件。

## 第一步：收集信息

需要以下信息。按优先级从三个来源获取：

**来源A — 参数**：从 `$ARGUMENTS` 中提取项目名和描述。
**来源B — 上下文**：如果对话中刚讨论了项目，从中提取目标、首要任务、约束、技术细节。
**来源C — 询问用户**：不确定的信息直接问用户。不要猜测。基于上下文的合理推断可以，凭空编造不行。

**必需信息**（创建前必须确定）：
- 项目名称（简短，用于文档标题）
- 项目描述（1-3句话）
- 第一阶段名称（初始工作叫什么）
- 第一个具体任务（先做什么——用于"车灯"部分）

**可选信息**（不确定就问，留空也可以）：
- 初始铁律
- 关键技术细节（工具、语言、路径）
- 已知子项目

## 第二步：检查已有文件

检查 CLAUDE.md 或其他 Doc Harness 文件是否已存在。如存在，覆盖前须警告用户。

## 第三步：创建5个文件

### 文件1：CLAUDE.md

```markdown
# [项目名] — 入口文档

**最后更新**: [今天日期]
**当前阶段**: [阶段名]（⏳）
**一句话状态（截至[今天日期]）**: [描述] — 项目刚初始化

---

## 恢复链

1. 读本文件 → 项目概述、铁律、操作规则
2. 读 `CURRENT_STATUS.md` → 当前工作、近期历史、下一步
3. 按需：`FILE_INDEX.md` → 查找文件
4. 按需：`WORKLOG.md` → 查找历史详情

## 项目概述

[将描述扩展为3-5行]

## 项目级铁律

[用户提供的规则，或默认：]
- "落笔为安"——重要信息必须保存到文件并注册
- [项目特定规则]

## 关键技术信息

[来自上下文，或："随项目进展填充"]

## 子项目（如有）

[如有则列出，否则删除本节]

---

## Doc Harness — 操作规则

[此处嵌入 operational_rules.md 的完整内容]
```

**重要**：读取 [operational_rules.md](operational_rules.md) 并将其完整内容嵌入上述 CLAUDE.md 中标记的位置。

```markdown

**当前阶段**: [阶段名]

---

## 近期已完成（车辙）

---

### 阶段目标
[本初始阶段要完成什么]

### 已完成步骤
（尚无）

---

## 下一步（车灯）

### 立即行动
1. [从收集的信息中提取的第一个具体任务]

### 后续规划
- [从上下文获取的更远期目标]

---

## 当前工作原则（驾驶手册）

- 遵循 Doc Harness 操作规则（见 CLAUDE.md）
- [来自上下文的阶段特定原则]
```

### 文件3：FILE_INDEX.md

```markdown
# FILE_INDEX — [项目名]

**最后更新**: [今天日期]

---

## 核心文档
- `CLAUDE.md` — 项目入口
- `CURRENT_STATUS.md` — 当前状态
- `FILE_INDEX.md` — 本文件
- `WORKLOG.md` — 工作历史
- `DOC_HARNESS_SPEC.md` — Doc Harness 规范（参考文档）
```

### 文件4：WORKLOG.md

```markdown
# WORKLOG — [项目名]

## 目录
| 阶段 | 时间 | 锚点 |
|------|------|------|
| （暂无已完成阶段）| | |
```

### 文件5：DOC_HARNESS_SPEC.md

读取 [spec.md](spec.md) 并将其内容写入项目目录的 `DOC_HARNESS_SPEC.md`。

如果 spec.md 不可访问，在 FILE_INDEX 中注明："DOC_HARNESS_SPEC.md — 尚未部署（CLAUDE.md 中的操作规则足以支持日常使用）"。

## 第四步：验证

- 5个文件都存在
- FILE_INDEX 列出了全部5个文件
- CURRENT_STATUS 有全部4个区域
- CLAUDE.md 包含嵌入的操作规则
- WORKLOG 有空目录表

报告："Doc Harness 已为 [项目名] 初始化。创建了5个文件。"
