## 第一章 首要原则与概述

### 1.1 首要原则："落笔为安"（Write It Down or Lose It）

Context中的一切信息都是临时的。无论多重要的分析结果、多精妙的设计思路，只要没有**写入文件**并**注册到FILE_INDEX**，就等于不存在。

- **落笔**：重要信息必须写入文件。不写 = 丢失。
- **为安**：写入的文件必须注册到FILE_INDEX。不注册 = 虽存在但不可发现 = 等同于丢失。

这条原则优先于一切效率考量。宁可多写一个文件，不可让信息只存在于context中。

### 1.2 什么是状态文档体系

状态文档体系是一个由四个核心文档组成的信息管理系统，服务于一个**项目**。

| 文档 | 角色 | 核心职责 | 更新频率 |
|------|------|---------|---------|
| **CLAUDE.md** | 门牌 | 项目入口、恢复起点、项目级铁律 | 阶段切换时 |
| **CURRENT_STATUS.md** | 行驶中的汽车 | 当前阶段活跃信息、近期历史摘要、下一步、阶段级原则 | 每个session |
| **FILE_INDEX.md** | 图书馆目录 | 按类别索引项目所有文件 | 文件变化时 |
| **WORKLOG.md** | 行车记录仪 | 永久保存所有阶段的完整工作详情 | 阶段切换时（批量） |

### 1.3 核心场景

| 场景 | 需求 |
|------|------|
| **全新Agent到达** | 读CLAUDE.md → CURRENT_STATUS → 知道做什么 |
| **Context compact后恢复** | 读CURRENT_STATUS → 从车身恢复到中断点 |
| **跨项目切换** | 读目标项目CLAUDE.md → 快速了解 |
| **工作过程中** | 实时更新CURRENT_STATUS车身 + 新文件注册FILE_INDEX |

### 1.4 不依赖外部记忆系统

状态文档体系完全自足。不依赖MEMORY.md、不依赖对话历史、不依赖任何Agent的私有记忆。任何Agent仅凭阅读文件即可完全恢复工作状态。

如果环境提供了记忆系统（如Claude Code的MEMORY.md），可以作为**补充加速**（提供额外上下文），但不作为恢复的必要条件。

## 第二章 CLAUDE.md — 入口文档

### 2.1 职责
- 告诉来者"这是什么项目"
- 提供恢复链（阅读顺序）
- 定义项目级铁律（永久有效，不随阶段变化）
- 指向其他三个核心文档
- 提供项目特有的稳定技术信息

### 2.2 不做什么
- 不记录详细当前状态（→ CURRENT_STATUS）
- 不罗列所有文件（→ FILE_INDEX）
- 不记录历史工作详情（→ WORKLOG）

### 2.3 模板

CLAUDE.md由三部分组成：项目特定信息 + 操作版规则（嵌入） + 完整版规范指针。

```markdown

> 🔒 **AGENT IDENTITY LOCK（代理身份锁定）**
>
> **你是 [项目名] 项目的代理。**
> [一句话描述本项目 agent 的职责。]
>
> 如果你对自己的身份有任何怀疑，立即停止操作并重新读取本段落。

---

**最后更新**: YYYY-MM-DD
**当前阶段**: [阶段名称]（⏳/⏸️/✅）
**一句话状态（截至YYYY-MM-DD）**: [简述]

## 恢复链

恢复链是任何 agent 或人类继续本项目工作的入门仪式，分两层。

### 必读（基线）

任何继续工作的人都必须读的文件。保持在 2–3 条。

1. 本文件（CLAUDE.md）
2. `CURRENT_STATUS.md`

### 任务条件读

只在工作匹配时读。随项目演化增减。

- 如要查找具体文件：读 `FILE_INDEX.md`
- 如要查历史阶段详情：读 `WORKLOG.md`
- [按需添加项目特定条目——如"做 X 时读 Y.md"]

### 元规则

- 恢复链必须**自含**：条目只指向项目内（或兄弟项目稳定路径的）文件。不依赖 agent 侧特性（memory 系统、对话历史、外部服务）——下一个 agent 可能没有这些。
- 恢复链是**活的**：有新类型工作出现时加条件读条目；旧类别退役时删除。阶段切换时审查。
```

- 操作规则部分一般不修改——它由创建时的规范版本决定

## 第三章 CURRENT_STATUS.md — 当前状态文档

### 3.1 职责与结构

CURRENT_STATUS 是项目的**活跃神经中枢**。把项目想象成一辆正在行驶的汽车，CURRENT_STATUS 就是驾驶舱视图。四个区域各有不同职责、不同更新节奏、不同比喻：

- **§1 车辙（近期已完成）** —— 后视镜里看到的。最近 2–3 个已完成的阶段，每个 3–5 行。足以回答"我们走过哪里"，但不必为细节去翻 WORKLOG。每条指向 WORKLOG 对应章节。滚动窗口：新阶段完成时，最老的车辙滑出。
- **§2 车身（当前工作）** —— 汽车本身，此刻所在之处。当前阶段的完整工作记录：阶段目标、每个已完成步骤（做了什么、创建了什么文件、关键决策）、未解决的问题。这是编辑最频繁的区域——每完成一个有意义的步骤都应落在这里。阶段切换时，车身内容迁入 WORKLOG（永久归档），车身清空以供下一阶段使用。
- **§3 车灯（下一步）** —— 前方被照亮的部分。立即行动（1–3 条具体下一步）加长期规划。Session 结束时，车灯应仍指向可执行之处，让下一个 agent（或你自己）能接上继续。每 session 更新。
- **§4 驾驶手册（当前工作原则）** —— 在*这条路*上驾驶*这辆车*的规则。阶段特定的工作准则（如"本阶段样本外验证"/"提交前运行 typecheck"）。随阶段生灭；切换时逐条审查（提升为铁律、保留、或移除——见 §6.2.2）。

汇总：

| 区域 | 比喻 | 内容 | 生命周期 |
|------|------|------|---------|
| §1 近期已完成 | 车辙 | 最近 2–3 个已完成阶段的简要总结（各 3–5 行） | 滚动窗口 |
| §2 当前工作 | 车身 | 当前阶段的全部详细工作记录 | 阶段结束时 → WORKLOG |
| §3 下一步 | 车灯 | 立即行动 + 后续规划 | 每 session 更新 |
| §4 当前工作原则 | 驾驶手册 | 本阶段适用的注意事项和原则 | 随阶段生灭 |

**比喻的意义**：四个区域回答 agent 到达时会问的四个不同问题——*我们走过哪里？*（车辙）、*现在在哪里？*（车身）、*接下来去哪里？*（车灯）、*在这条路上怎么开？*（驾驶手册）。物理上分开四个区域，是为了防止它们相互溶解——状态文档最常见的失败模式就是退化为一段散文叙述。

### 3.2 模板

```markdown
# CURRENT_STATUS — [项目名]

**最后更新**: YYYY-MM-DD
**当前阶段**: [阶段名称]

---
```

### 3.3 车身的更新粒度

**关键规则：为compact做好准备。** 不要假设当前session能完整结束。

- 每完成一个**有意义的步骤**后，更新车身。"有意义"= 如果此刻丢失了context，下一个Agent需要知道这一步已完成。
- 车身内按**步骤**（非session）组织。一个session可能完成多个步骤，一个步骤也可能跨多个session。
- **车身长度建议**：控制在约200行以内。如果超过，考虑是否应触发阶段切换。

### 3.4 更新规则
- 每个session至少更新一次（车身的已完成步骤 + 车灯的下一步 + 最后更新日期）
- 创建新文件时，在车身中记录"创建了xxx"，同时注册FILE_INDEX

## 第四章 FILE_INDEX.md — 文件索引文档

### 4.1 职责
按**类别**索引项目所有文件。回答"项目里有哪些文件、是什么、在哪里"。

### 4.2 组织原则
- 按类别（##标题）分组，**不按时间排列**
- 类别是项目特定的，可以随时新增或重组
- 每个条目：文件路径 + 一句话说明
- 如果一个文件横跨两类，只放在主类别中，另一类别下用"→ 另见 §[主类别]"指向

### 4.3 递归子索引

当某类别的文件数量超过**10个**时，建议在该类别文件夹中创建子FILE_INDEX.md。超过**20个**时，必须创建。

父FILE_INDEX对该类别只保留一行：
```markdown
- `notes/findings/` → 见 [`notes/findings/FILE_INDEX.md`](notes/findings/FILE_INDEX.md)
  （Ch04/Ch05研究发现文档，共38个，按章节分类）
```

子FILE_INDEX头部标注父索引：
```markdown
# FILE_INDEX — [类别名]
**父索引**: [`../../FILE_INDEX.md`](../../FILE_INDEX.md)
```

递归深度限制为**2层**（父→子→孙）。如果需要更深，说明项目结构过于复杂，应考虑重组文件夹。

### 4.4 批量文件创建

当一个session创建大量同类文件（如一次性生成20个分析报告），不必逐一注册。可以：
1. 在FILE_INDEX中注册**文件夹**，标注文件数量和命名规则
2. 如果文件数量>10，创建子FILE_INDEX.md
3. 在CURRENT_STATUS车身中记录"创建了N个文件于xxx/，命名规则为..."

### 4.5 模板

```markdown
# FILE_INDEX — [项目名]文档目录

**最后更新**: YYYY-MM-DD

---

## 核心状态文档
- `CLAUDE.md` — 项目入口
- `CURRENT_STATUS.md` — 当前状态
- `FILE_INDEX.md` — 本文件
- `WORKLOG.md` — 工作历史

## [类别1名称]
- `path/file1.md` — 说明
- `path/file2.md` — 说明

## [类别2名称]
- `path/folder/` → 见 [`path/folder/FILE_INDEX.md`]（N个文件，简述）

## [类别3名称]
- `path/file3.py` — 说明
```

### 4.6 更新规则
- 创建新文件时注册
- 删除/归档文件时移除
- 不需要每个session更新——只在文件系统发生变化时

## 第五章 WORKLOG.md — 工作日志文档

### 5.1 职责
永久保存所有阶段的**完整**工作详情。内容来源于CURRENT_STATUS阶段切换时的转移。

### 5.2 头部目录（TOC）

WORKLOG在文件顶部维护一个目录，方便在长文档中快速定位：

```markdown
# WORKLOG — [项目名]

## 目录
| 阶段 | 时间 | 页内锚点 |
|------|------|---------|
| 阶段5: 格式修正 | 2026-03-29~03-30 | [→](#阶段5-格式修正) |
| 阶段4: 文献补强 | 2026-03-27~03-29 | [→](#阶段4-文献补强) |
| 阶段3: Ch5重建 | 2026-03-25~03-27 | [→](#阶段3-ch5重建) |
| ... | | |

---

## 阶段5: 格式修正
[详细内容...]

## 阶段4: 文献补强
[详细内容...]
```

### 5.3 内容格式

每个阶段是一个顶级章节（##），包含：

```markdown
## 阶段N: [名称]（YYYY-MM-DD ~ YYYY-MM-DD）

### 阶段总结
[阶段结束时添加的3-5行总结]

### 详细记录
[从CURRENT_STATUS车身区原封不动复制的内容]
```

### 5.4 修改规则

- **允许**：在已有阶段的开头**追加**总结或勘误注释
- **允许**：在顶部目录中添加新条目
- **禁止**：修改或删除从CURRENT_STATUS复制来的原始内容
- 事实性错误以勘误方式标注：`[勘误(YYYY-MM-DD): 实际为4087篇，非4090篇]`

### 5.5 归档

WORKLOG 只增不改，必然增长。一个 2000 行的 WORKLOG 本身就是负担——任何被要求"查历史"的 agent 一次取走全部就烧光了 context。归档就是让 WORKLOG.md 本身保持可读，同时在磁盘上完整保留历史。

**触发**（软阈值）：WORKLOG.md 超过**约 1000 行**。用行数而非阶段数衡量——各项目阶段密度相差 10 倍，行数才是可靠单位。

**归档流程**：

1. 决定切点：`WORKLOG.md` 保留最近 **3 个阶段**，更早的迁出。
2. 创建 `WORKLOG_ARCHIVE_<YYYY-MM-DD>.md`，日期为**归档事件发生的日期**（即今天——你执行归档的日期），而非固定周期分箱。每次归档事件生成一个独立文件，使单个归档天然有上限（单次归档最多搬走 ~1000 行阈值触发时的更早部分）。多个归档文件按时间先后累积。*为何采用"按事件"而非"按固定周期"：如果按季度命名，活跃项目一个季度写 5,000 行，整 5,000 行会灌入同一个"季度归档"文件，归档本身就失去意义。按事件命名，每个归档文件的上限天然由触发阈值决定。*
3. 将更早的阶段章节（含目录表条目）迁入归档文件。归档文件结构与 WORKLOG.md 相同（顶部目录 + 阶段章节）。
4. 在归档文件顶部（其自身目录表之上）添加**反向指针**：
   ```
   > 属于 WORKLOG 系列。当前 WORKLOG：`WORKLOG.md`。其他归档：`WORKLOG_ARCHIVE_2025-12-03.md`, `WORKLOG_ARCHIVE_2026-02-14.md`, ...
   ```
   其他归档按归档事件时间先后顺序列出。
5. 在 `WORKLOG.md` 标题下方紧接着添加（或更新）**前向指针**：
   ```
   > 更早的历史归档于：`WORKLOG_ARCHIVE_2025-12-03.md`, `WORKLOG_ARCHIVE_2026-02-14.md`, `WORKLOG_ARCHIVE_2026-04-19.md`, ...
   ```
   新归档追加到现有列表末尾，保持时间先后顺序。
6. 在 FILE_INDEX 的 `## 历史归档` 类别下注册**新的**归档文件（若类别缺失则创建）。已有归档条目保持不动，只需添加本次归档的新条目。
7. **在 CURRENT_STATUS 中记录归档事件**。在当前阶段"已完成步骤"车身区添加一行：
   ```
   - (YYYY-MM-DD) 归档 WORKLOG：阶段 N–M 迁入 `WORKLOG_ARCHIVE_<YYYY-MM-DD>.md`；活跃 WORKLOG 现保留最近 3 个阶段。
   ```
   这让下一会话的 agent 仅读 CURRENT_STATUS 就能看到历史迁移发生——无需 diff WORKLOG。

**Git 提交规范**：归档动作作为**单次原子提交**，提交信息 `Archive WORKLOG <YYYY-MM-DD> (keep most recent 3 phases)`。**不要**将归档动作与无关工作合并——未来 agent 追溯历史时应能干净地识别归档事件。单次提交内的暂存顺序：新增归档文件、修改 WORKLOG、修改 FILE_INDEX、修改 CURRENT_STATUS——全部一个提交。

**非目标**：归档不是删除。归档文件是永久记录，除勘误外不再修改，永不删除。

**何时可跳过**：阶段密度小（如每阶段 ~20 行）的项目可能永远不触发。这没问题——阈值只捕捉确实需要的项目。

## 第六章 信息流动规则

### 6.1 日常工作流

```
读CLAUDE.md → 读CURRENT_STATUS → 开始工作
工作中：
  ├── 每完成一个有意义的步骤 → 更新CURRENT_STATUS车身
  ├── 创建新文件 → 注册FILE_INDEX + 记录在CURRENT_STATUS车身
  └── session结束 → 确认车身反映了全部工作 + 更新车灯 + 更新日期
```

### 6.2 阶段切换流（五步操作）

**触发条件**（满足任一即可触发）：
- 工作目标发生质变（如从"写作"转入"审读"）
- 用户明确宣布进入新阶段
- 项目暂停（超过3天无工作）
- 车身已超过200行

**不触发**：同一目标内的子任务切换（如文献补强中从Ch4到Ch5）

**五步操作**（按此顺序执行）：

```
Step 1（保护性操作，最先执行）:
  将CURRENT_STATUS车身全部内容复制到WORKLOG顶部（目录之后），
  添加阶段总结。
  → 此步完成后，即使后续步骤被中断，也不会丢数据。

Step 2:
  在CURRENT_STATUS车辙区最顶部插入新摘要（3-5行总结本阶段）。
  标注"详细记录见WORKLOG.md §阶段N"。

Step 3:
  如果车辙已有3个摘要，删除最老的。

Step 4:
  清空车身。填入新阶段的目标。
  应用驾驶手册审查仪式（见下文 §6.2.2）。

Step 5:
  更新CLAUDE.md的"当前阶段"和"一句话状态（截至日期）"。
  更新WORKLOG的目录表。
```

**中断安全**：Step 1最先执行（将内容复制到WORKLOG），确保即使compact在Step 2–5之间发生，数据已安全保存。中断状态的检测与修复流程见 §6.3.1。

#### 6.2.1 各步骤的失败模式

每个步骤有其典型的失败签名。下表供诊断项目异常状态时参考（亦见 §6.3.1）：

| 步骤 | 失败签名 | 修复 |
|------|---------|------|
| 1 | WORKLOG 无新条目；车身仍满 | 数据尚无风险——重试 Step 1。 |
| 1（部分） | WORKLOG 有详细块但无阶段总结 / 无 TOC 行 | 从车身补写总结；补 TOC 行。 |
| 2 | CURRENT_STATUS 车辙缺新摘要；WORKLOG 已有条目 | 从 WORKLOG 复制摘要到车辙。 |
| 3 | 车辙 > 3 条 | 删除最老的（已永久存于 WORKLOG）。 |
| 4 | 车身仍为旧阶段内容；新阶段目标未写 | 清空；写目标。 |
| 4（部分） | 驾驶手册未审查 | 应用 §6.2.2 仪式。 |
| 5 | CLAUDE.md"当前阶段"仍为旧；或 WORKLOG TOC 缺新行 | 两处都更新。 |

§6.3.1 mid-transition 检测触发的修复应按下游依赖顺序执行，直到项目状态重新自洽为止。

#### 6.2.2 驾驶手册审查仪式

每次阶段切换的 Step 4，对驾驶手册中的每一条原则执行以下审查。以此替代临时判断。

对驾驶手册中每一条原则 P：

1. **P 是否已标注 `[DONE]` / `[已满足]`？**
   → 既然阶段即将结束（切换时点），**删除 P**。已满足原则不跨阶段携带。

2. **P 是否已连续 ≥3 个阶段出现，且语义意图未变？**
   这里"未变"指*规则还在说同一件事*，并非字面文本完全相同——为清晰而改写（规则 4）不重置计数器。如有疑问：若违反其中一条也必然违反另一条，则两条原则意图相同。
   → **提升候选**：考虑提升为 CLAUDE.md 铁律。确认方式：问"未来任一阶段违反 P 会造成真实问题吗？"如是则提升。提升需记录在本次阶段切换的 WORKLOG 条目中。

3. **P 是否在回应已不存在的约束？**（如项目已迁离库 X 后还写着"始终使用库 X"）
   → **删除 P**。

4. **P 是否适用于新阶段预期的工作？**
   → **保留 P**。必要时为清晰而改写（但意义不变）。

5. **P 是否仅适用于新阶段的部分子任务？**
   → **保留 P**，但考虑添加范围标注。

然后添加新阶段特有的新原则。每阶段 1–4 条为宜；驾驶手册超 6 条通常表示过拟合——精简。

### 6.3 Context compact 恢复流

```
Compact 发生
  → 读 CLAUDE.md → 了解项目和恢复链
  → 读 CURRENT_STATUS.md：
      车辙 → "之前做完了什么"
      车身 → "当前做到哪里了"
      车灯 → "接下来做什么"
      驾驶手册 → "工作中注意什么"
  → 执行 §6.3.1 Mid-Transition 检测（见下）
  → 继续工作（如果用户在场，先确认下一步是否有变）
```

#### 6.3.1 Mid-Transition 检测与修复

阶段切换可能在 Step 1 与 Step 5 之间被中断（compaction、崩溃、手工中断）。症状取决于哪一步未完成。每次 session 开始都跑一下这个三值一致性检查：

**待比较的三个值**：
- **A** — CLAUDE.md 中"当前阶段"字串。
- **B** — WORKLOG TOC 最新条目（TOC 表头之后的首行）。
- **C** — CURRENT_STATUS 车辙中最新的阶段摘要（"近期已完成"下第一个 `###` 标题）。

**一致性规则**：

| A | B | C | 状态 | 行动 |
|---|---|---|------|------|
| 阶段 N | 阶段 N（旧） | 阶段 N−1 最新 | ✅ 正常阶段中 | 继续工作。 |
| ⏸️ | 任意 | 任意 | 已暂停——可能处于 mid-transition | 应用 §6.4 恢复规则；检测在那里继续。 |
| 阶段 N+1 | 阶段 N | 阶段 N−1 最新 | ❌ Step 1 未完成（阶段标题提前更新但 WORKLOG 未写） | **此情形不应发生**，若 agent 遵守五步顺序即可避免。一旦检测到：保护数据——重新读 CURRENT_STATUS 车身，当作仍在阶段 N，立即执行 Step 1。 |
| 阶段 N | 阶段 N（新） | 阶段 N−1 最新 | ❌ Step 2 未完成（WORKLOG 因 Step 1 写入了阶段 N 新条目但车辙未更新） | 从 WORKLOG 新条目复制阶段摘要到 CURRENT_STATUS 车辙顶部；若车辙此时超过 3 条，执行 Step 3。再继续 Step 4。 |
| 阶段 N | 阶段 N（新） | 阶段 N−1 最新，车辙 >3 | ❌ Step 3 未完成 | 删除最老的车辙条目。继续 Step 4。 |
| 阶段 N（旧） | 阶段 N（新） | 阶段 N（新）最新 | ❌ Step 5 未完成（WORKLOG 与车辙已前进但 CLAUDE.md 仍为旧阶段） | 更新 CLAUDE.md"当前阶段"和"一句话状态（截至日期）"；若缺 TOC 行则补上。 |
| 其他不一致 | — | — | ⚠️ 模糊 | 在车身写诊断：`#### Mid-transition detected (YYYY-MM-DD)`。用户在场 → 询问；否则 → 按 WORKLOG 所指方向完成切换（WORKLOG 为"本应发生什么"的权威来源）。 |

**关于 B 列"阶段 N"的脚注**："阶段 N（旧）"表示 B 指向*当前*阶段的 WORKLOG 条目（阶段 N 开始时写入）；"阶段 N（新）"表示 B 指向一条*新写入*的、标志 Step 1 完成（即从阶段 N−1 切换出去）的新阶段条目。区分的关键在与 C 的关系：当 C 是阶段 N−1（健康前驱）时，B = 阶段 N 是稳定的阶段中条目；当 C 是阶段 N−1 *且* B 位置有一条新阶段 N 行，其内容与 CURRENT_STATUS 车身不符时，说明 Step 1 写了 B 但其余未落地。Agent 通过比较阶段名字串与日期字段来实现本表，而非依赖语义推断。

**零进展 Step 1 中断**：此表无法检测*尚未发生任何写入*的中断（A、B、C 均仍为切换前状态）。这并非数据丢失场景——根本还没开始切换——但若阶段目标已接近切换触发条件，下一个 agent 应慎重考虑是否继续延伸车身。如不确定，agent 应检查 §6.2 的触发条件在 compaction 前是否已经满足；若是，应主动启动新一轮阶段切换，而非继续沿用旧阶段。

**任一修复动作后**：在 CURRENT_STATUS 车身加一行记录修复（如 `#### 在 compact 恢复中自动修复阶段切换 Step 2（2026-04-19）`）。每次 compact 恢复只加一行；后续编辑属于正常工作。

**为何成立**：Step 1（数据入 WORKLOG）是保护性的——一旦 Step 1 完成，真实状态不会丢失。Step 2–5 是事务性记录，可从 WORKLOG 状态确定性地补完。真正危险的情况（表中第一行）只要 agent 遵守五步顺序就不会出现；检测只是为了偏执保险。

### 6.4 项目暂停与恢复

#### 有序暂停

1. 执行阶段切换五步操作（将当前工作存入 WORKLOG）。
2. 在 CURRENT_STATUS 车身写：`项目暂停于 YYYY-MM-DD。暂停原因：...`
3. 更新 CLAUDE.md 状态为 ⏸️，刷新一句话状态。
4. 在车身暂停说明中写明任何**时效性上下文**，即会让车灯在恢复时失效的因素（如"依赖 X 交付物，预期 2026-05-15；在按车灯行动前先确认 X 的当前状态"）。

#### 紧急暂停 / 阶段切换途中暂停

如暂停本身发生在阶段切换途中（罕见但可能），**不要**试图先完成切换——时间压力下那样做有数据丢失风险。应：

1. 在最安全可编辑的文档（通常是 CURRENT_STATUS 车身；如车身也在不自洽状态，退回到 CLAUDE.md 一句话状态）写：`项目暂停于 YYYY-MM-DD（阶段切换途中，进行至 Step N）`。
2. 将 CLAUDE.md 状态置为 ⏸️。
3. 恢复时，§6.3.1 mid-transition 检测会识别中断状态并先修复，再恢复正常工作。

#### 恢复（用户在场）

1. Agent 读 CLAUDE.md → 看到 ⏸️。
2. 读 CURRENT_STATUS → 车辙有最后阶段摘要，车身有暂停说明。
3. 读 WORKLOG 最新阶段 → 获取暂停前完整工作详情。
4. **关键**：车灯的"下一步"可能已过期。按车灯执行前先与用户确认，特别留意暂停时记录的时效性上下文。
5. 将 CLAUDE.md 状态从 ⏸️ 翻转为 ⏳；刷新一句话状态。

#### 恢复（用户不在场——自动恢复）

当 agent 被 compact 回到一个暂停项目，且无用户直接输入时触发。

1. Agent 读 CLAUDE.md → 看到 ⏸️。
2. 读车身中暂停说明，记录暂停日期。
3. **先执行 §6.3.1 mid-transition 检测。** 如检测到 mid-transition 状态，**在做恢复决定之前先修复**。暂停本身可能发生于阶段切换途中（见上文紧急暂停路径）；未修复便恢复有可能基于不自洽状态行动。
4. §6.3.1 返回一致状态后，**比较暂停日期与今日**：
   - **≤7 天**（今日、1、2、3、4、5、6、7 天前）：大概率可恢复。对照时效性上下文（若有）核查车灯。若一切看来仍成立：在车身写 `#### 自动恢复于 YYYY-MM-DD（用户未在场）`，将 CLAUDE.md 翻转为 ⏳，按车灯继续。
   - **8–30 天**：车灯可疑。**不要**静默执行车灯。在车身写 `#### 自动恢复已尝试 YYYY-MM-DD——等待用户确认`。CLAUDE.md 保持 ⏸️。下次与用户交互时提出问题。
   - **>30 天**：永不自动恢复。等待用户明确指示。

## 第八章 工作原则的两级管理

### 8.1 两类原则

| 类型 | 存放位置 | 生命周期 | 示例 |
|------|---------|---------|------|
| **项目级铁律** | CLAUDE.md"铁律"区 | 与项目同寿 | "不得直接读取论文.md原文" |
| **阶段级原则** | CURRENT_STATUS"驾驶手册"区 | 随阶段生灭 | "本阶段每次修改LaTeX后必须编译验证" |

### 8.2 原则的升降级

- 阶段切换时，检查驾驶手册中是否有原则**应提升为项目级铁律**
- 如果一条原则在连续3个阶段都出现，大概率应该升级
- 升级：从驾驶手册移到CLAUDE.md铁律区
- 降级很少发生——铁律一旦确立通常不撤销

### 8.3 铁律的管理

铁律存放在CLAUDE.md中，与项目同寿。可以在阶段切换时新增铁律（从驾驶手册中提升）。铁律一般不删除——如果某条铁律的前提条件不再适用，标注"[已不适用: 原因]"但保留原文，以供历史参考。

### 9.1 核心状态文档

### 9.2 叶子文档

| 状态 | 含义 | 标记方式 |
|------|------|---------|
| 活跃 | 正在使用或仍有效 | 默认（不标记）|
| SUPERSEDED（保留） | 已被替代，但仍有参考价值 | 文件首行：`⚠️ SUPERSEDED BY [新文档]`。在FILE_INDEX中移至 `## 已替代` 类别。|

## 第十章 子项目与项目群

### 10.1 子项目
子项目拥有完整的四文档体系，递归适用本规范。父项目FILE_INDEX中注册子项目入口。

### 10.2 项目群
多个独立项目共享父目录时，父目录可有一个轻量级CLAUDE.md（仅导航表，无CURRENT_STATUS/FILE_INDEX/WORKLOG）。

```markdown
# [工作区] — 项目导航

## 活跃项目
| 项目 | 入口 | 状态 |
|------|------|------|
| 项目A | [CLAUDE.md](A/CLAUDE.md) | ⏳ |
| 项目B | [CLAUDE.md](B/CLAUDE.md) | ⏸️ |

## 共享基础设施
| 工具 | 路径 | 说明 |
|------|------|------|

## 归档项目
| 项目 | 路径 | 完成日期 |
|------|------|---------|
```

Agent到达项目群时，通常由用户指令决定进入哪个子项目。

## 第10.3节 适应已有项目

**核心原则：状态文档体系适应项目，而非项目适应体系。**

### 中途引入

现实中大多数引入发生在项目进行过程中——项目已有文件、可能已有状态文档、也有一定的工作历史。引入流程如下：

1. **先理解项目。** 在创建状态文档之前，仔细阅读所有现有的状态/说明文档。了解项目目标、当前状态、目录结构以及已有的文档。

2. **创建四个核心文档。** CLAUDE.md、CURRENT_STATUS.md、FILE_INDEX.md、WORKLOG.md——填入项目的实际当前状态，而非空白模板。

3. **处理已有的状态文档。** 大多数应标记为 SUPERSEDED 但保留：
   - 在文件首行添加 `⚠️ SUPERSEDED BY [新文档]`
   - 在 FILE_INDEX 中移至 `## 已替代` 类别
   - 注明保留原因（如"包含详细的session记录"或"含有风格指南可供参考"）

4. **从历史记录重建 WORKLOG。** 利用现有日志、状态文档和git历史来构建阶段记录。重建的部分应标注："（根据[来源文档]重建）"。目标是有用的历史，而非完美的精确度——实时记录的细节度始终更高。

5. **保留已有的高质量索引。** 如果某个子目录已有维护良好的索引（如含100+条目和状态标签的研究分析INDEX.md），FILE_INDEX.md应当**指向它**而非重建。添加一行即可：`→ 见 [subdir/INDEX.md]（N个条目，覆盖[范围]）`。

### 大型目录的子索引指引

对于超过20个文件的目录，创建子FILE_INDEX.md。初次创建大型子索引（100+文件）时，可通过文件系统扫描自动生成初始结构，然后在可能的情况下进行人工验证和标注。

## 第十一章 Agent行为规范

### 11.1 Session开始——应用恢复链

按项目 CLAUDE.md 中定义的恢复链执行：

**第 0 步 — 身份锚定**（在读任何文件之前）：
读 CLAUDE.md 顶部的 AGENT IDENTITY LOCK。确认：
"我是 [项目名] 的代理。我的职责是 [锁中描述的角色]。"

1. 按顺序读**必读**层（CLAUDE.md → CURRENT_STATUS.md，通常 2–3 个文件）。
2. 扫描**任务条件读**层。只读条件匹配当前工作的条目。
3. 如果用户在场 → 确认"下一步"是否仍然正确。
4. 如果是 compact 恢复（用户不在场）→ 直接按车灯继续。

不要在启动时无条件读完所有任务条件读条目——那会让两层切分失去意义。如果你发现某条条件读条目每次都读，它其实属于必读层，应显式提升。

### 11.2 工作过程中

1. 每完成一个**有意义的步骤** → 更新CURRENT_STATUS车身
2. 创建新文件 → **同时做两件事**：(1)注册FILE_INDEX (2)记录在CURRENT_STATUS车身
3. **"落笔为安"**：重要分析结果、设计思路、决策理由 → 写入文件 + 注册
4. **关注剩余 context**（如运行环境暴露该信息）：context 压缩等同于非自愿的 session 结束。如果环境报告 context 使用率（百分比或 token 数），将**剩余不足**（~<20%）视为立即触发——在下一次工具调用前更新 CURRENT_STATUS；如果车身已积累**大量未保存的工作**，立即考虑阶段切换。别等那个可能永远不会到来的"有意义的步骤"。

   **"大量未保存的工作"——具体阈值**（任一成立即可）：(a) 距上次车身刷新已完成 ≥3 个 `####` 步骤；(b) 距上次刷新车身已新增 ≥50 行；(c) 车身已 ≥100 行（阶段切换 200 行阈值的一半）。若 (a)–(c) 均不成立，只刷新即可——不必从单薄的车身强行触发阶段切换。

### 11.3 Session结束

1. 确保CURRENT_STATUS车身反映了本session的全部工作
2. 更新车灯的"下一步"
3. 更新CURRENT_STATUS的"最后更新"日期
4. 更新CLAUDE.md的"一句话状态（截至日期）"
5. 检查：本session创建的文件是否都注册了FILE_INDEX
6. 检查：有没有重要信息只存在于context中？→ 落笔为安
7. 如发生了阶段切换：检查 WORKLOG 长度——超过约 1000 行？→ 按 §5.5 触发归档。

### 11.4 阶段切换

执行第六章§6.2的五步操作。

### 11.5 状态同步（`/doc-harness sync`）

**目的**：主动修复文档漂移。与 `check`（只读诊断）不同，`sync` 修改文件以使其重新与现实对齐。

**触发条件**：
- 用户显式请求同步
- 智能体在 session 结束检查清单中检测到显著漂移
- 长时间项目暂停之前
- 在正常智能体工作流之外进行批量文件操作之后

**模式**：
- **`interactive`**（默认）：同样的修复，但在阶段切换、归档或从上下文中创建新原则文档之前询问用户。
- **`auto`**：执行所有安全修复（文件注册、日期刷新），无需询问。若车身 ≥200 行或 WORKLOG ≥1000 行，自动执行阶段切换或归档（spec §6.2 和 §5.5）。

**执行步骤**：详见 `sync.md` 的完整分步流程（漂移扫描 → 自动修复 → 阶段切换检查 → 归档检查 → 原则提取）。

**与 `check` 的关系**：`check` 报告；`sync` 修复。运行 `check` 以全面了解健康状况；运行 `sync` 以修复漂移。`sync` 不执行原则反思——那仍是 `check` 的领域。

### 11.6 上下文冲刷（`/doc-harness flush`）

**目的**：上下文压缩或会话结束前的紧急保存。包含 `sync` 的一切功能，加上强制性地将上下文中的重要信息提取到文档中。

**核心保证**：成功冲刷后，全新智能体阅读恢复链应能恢复到**仿佛上下文从未被压缩**的状态——知道*存在哪些信息*以及*到哪里去找*。

**触发条件**：
- 用户显式请求冲刷（"保存一切"、"为压缩做准备"）
- 上下文使用率很高，用户希望确保零丢失
- 已知上下文即将重置之前
- 长时间高密度会话之后，大量上下文累积

**模式**：
- **`interactive`**（默认）：向用户提议每次提取，供其批准或编辑路径。
- **`auto`**：使用启发式规则按类型对上下文信息进行分类并路由到适当文档（`notes/`、`design/`、`PHILOSOPHY.md` 等），无需询问。

**执行步骤**（五阶段）：
1. **阶段 A**：以相同模式运行 `sync`。
2. **阶段 B**：上下文盘点——智能体自我扫描上下文，按类型分类条目，跳过已持久化项（在车身或 FILE_INDEX 中）。
3. **阶段 C**：写入与注册——创建或追加文件，注册到 FILE_INDEX，在车身中记录。
4. **阶段 D**：验证——智能体模拟全新到达，确认所有提取信息均可发现。
5. **阶段 E**：冲刷标记——在 CURRENT_STATUS 车身中记录冲刷事件。

**阶段 B 和阶段 C 不可跳过**。若盘点结果为零个可提取条目，智能体仍必须产出显式的**空扫描报告**，记录扫描范围与排除依据。悄悄跳过阶段 B 是冲刷失败——输出与 `sync` 无法区分，必须纠正。

**自动模式启发式**：信息"重要"的标准是：用户强调过、是决策/理由、产生它需要显著 effort、或丢失会导致困惑。目标路径遵循现有项目约定；仅在需要时创建标准文件夹（`notes/`、`design/`、`lessons/`、`data`）。

**完整流程**：见 `flush.md`。

### 11.7 信息召回（`/doc-harness recall [query]`）

**目的**：从项目 Doc Harness 文档层次结构中检索信息。随着项目增长，信息分散在数十个已注册文件中；`recall` 提供一套系统化的检索协议，遵循 Recovery Chain 的层次——从高层面摘要到低层面细节。

**与其他命令的关系**：`recall` 是**只读**的（像 `check`，不像 `sync`/`flush`）。它不修改文件。如果在 recall 过程中注意到漂移，在输出中报告但不修复——那是 `sync` 的领域。

**查询类型**（智能体分类查询，然后搜索适当层级）：

| 类型 | 模式 | 主要层级 | 示例 |
|------|------|----------|------|
| **A. 状态 / 计划** | "下一步是什么？"、"当前状态？" | 第 0–1 层 (CLAUDE.md → CURRENT_STATUS) | "认证的计划是什么？" |
| **B. 历史 / 决策** | "我们为什么..."、"什么时候..." | 第 1–2 层 (CURRENT_STATUS → WORKLOG) | "为什么选 PostgreSQL？" |
| **C. 文件 / 主题查找** | "找关于...的文件" | 第 3–4 层 (FILE_INDEX → 文件) | "找缓存文档" |
| **D. 跨文档综合** | "总结..."、"关于...的一切" | 第 1–4 层 (全部) | "所有认证讨论" |

**分层搜索协议**（自上而下；得到回答后提前停止）：

- **第 0 层**: CLAUDE.md — 铁律、概览、恢复链
- **第 1 层**: CURRENT_STATUS.md — 轮胎印、车体、车灯、驾驶手册
- **第 2 层**: WORKLOG.md — 历史阶段摘要（TOC → 跳转到阶段）
- **第 3 层**: FILE_INDEX + 子索引 — 目录扫描，递归进入 sub-FILE_INDEX.md
- **第 4 层**: 单个已注册文件 — 描述不足时全文阅读

**搜索规则**：
1. 只搜索已注册文件（在 FILE_INDEX 或子索引中）。未注册文件不可见。
2. 尊重恢复链优先级：摘要优先于原始文件。
3. 子索引是一等公民：递归进入 sub-FILE_INDEX.md。
4. 每个声明都需引用文件和章节/行号。
5. "未找到" 是可接受的——不要幻觉。
6. 只读：从不修改文件。
7. 提前停止：如果第 1–2 层已回答问题，不要阅读第 4 层。
8. 范围限制：如果 >20 个文件匹配，总结 FILE_INDEX 并询问用户深入阅读哪些类别。

**完整流程**：见 `recall.md`。

## 第十二章 反模式

| 反模式 | 正确做法 |
|--------|---------|
| 同一数字在多个文档中硬编码 | 只在CURRENT_STATUS中维护，其他文档引用 |
| 被替代的文档不标注 | 首行添加SUPERSEDED标记 |
| 多条恢复链 | 恢复链只在CLAUDE.md中定义 |
| 恢复链依赖 agent 侧特性（memory / 对话历史 / 外部服务）| 恢复链必须自含——只指向项目内文件 |
| 恢复链必读层超过 3 条 | 多出的降级到任务条件读；必读层保持精简 |
| 创建文件不注册 | 创建即注册FILE_INDEX |
| CURRENT_STATUS无限膨胀 | 阶段切换时详情转移到WORKLOG |
| WORKLOG 超过约 1000 行未处理 | 按 §5.5 归档 |
| 信息只存在于context中 | "落笔为安"——写入文件并注册 |
| 阶段级原则变成永久遗产 | 阶段切换时审查：升级为铁律或移除 |
| WORKLOG无目录 | 头部维护TOC表 |

## 第十三章 可选长期文档

核心四文档（CLAUDE / CURRENT_STATUS / FILE_INDEX / WORKLOG）面向中短期工作流：当前阶段是什么、做完了什么、文件在哪、历史怎样。有两类信息不适合放进这四份里。Doc Harness 为它们提供两个**可选**文档，只在项目真正积累相应内容时才使用。

这些文档是可选项。不需要时就不要创建。

### 13.1 PARKING_LOT.md — 暂缓事项

**目的**：想做但现在做不了的具体事。没有它，这类事项要么挤在 CURRENT_STATUS 车灯里（错，它们不是"下一步"），要么被新工作覆盖而遗忘。

**何时创建**：当某个具体事项正从当前视野移出、但又不应丢失。典型触发：阶段切换时，某条驾驶手册原则或下一步事项不再属于*当下*，但在已知条件下未来会重新相关。

**条目格式**：每条记录**要做什么**、**为何推迟**、**复活的前置条件**、**何时复查**。

```markdown
## 条目

### [事项标题] — 推迟于 YYYY-MM-DD
- **要做**：[具体行动或目标]
- **为何推迟**：[原因——如"依赖 X，X 要到 Q3 才发布"]
- **复活前置条件**：[什么为真时可以恢复]
- **复查**：[日期或事件——如"下次阶段切换时复查"或"2026-09-01"]
```

**生命周期**：前置条件满足时，要么 (a) 将事项迁回 CURRENT_STATUS 车灯，从 PARKING_LOT 划除；要么 (b) 保留条目并标注决定——如"[已复活 2026-09-15 → 现为活跃]"或"[已不再相关 2026-09-15 因为 X]"。不得悄悄删除。

**与恢复链的接口**：加入任务条件读层，如 `- 规划新阶段时：读 PARKING_LOT.md（有无可复活的暂缓事项？）`。

### 13.2 PHILOSOPHY.md — 来自实践的原则

**目的**：记录本项目具体实践催生的、可推广的原则、教训、洞察。这类内容比铁律更宽（铁律管*本项目*的操作），比驾驶手册更久（驾驶手册管*本阶段*）。

**设计原理**：通用的智慧来自具体的实践。项目是炉子，原则是从中出来的东西。禁止项目记录自己的哲学，等于切断原则与其诞生地的联系。PHILOSOPHY.md 就是这个捕获层。

**何时创建**：在项目工作中浮现一条洞察、你怀疑它可推广超出当下情境——一条可复用的教训、一个值得记住的设计权衡、一个不想重蹈的错误模式。

**条目格式**：每条陈述原则、催生它的实践、首次记录日期。

```markdown
## 原则

### [简短原则名] — 首次记录 YYYY-MM-DD
**原则**：[一句话陈述]

**源于**：[催生它的具体实践/事件/观察——具体说出*发生了什么*导致这条教训]

**范围**：[适用在哪——仅本项目？可能更广？已确认更广？]
```

**跨项目共享（可选）**：当一条原则被证明适用于本项目之外，可复制到兄弟项目自己的 `PHILOSOPHY.md`，或（若项目群存在一个父级导航 CLAUDE.md——见第十章）在那里引用。复制/引用**不移除**本项目 `PHILOSOPHY.md` 中的条目——每个采用该原则的项目各自保留诞生记录，本文件记录*本项目*如何得出这条原则。共享的条目可标注 `[亦被 <项目> 采用 YYYY-MM-DD]` 以便追溯。

**生命周期**：原则可以勘误精修（`[精修 YYYY-MM-DD: ...]`）或标注被替代（`[YYYY-MM-DD 被替代，新表述为: ...]`），但条目不删除。PHILOSOPHY.md 读得少，但应完整保留。

**与恢复链的接口**：用于反思性决策，加入任务条件读层，如 `- 做重大架构决策时：读 PHILOSOPHY.md`。

## 第十四章 可选的跨项目通信（inbox/outbox）

当一个项目与其他项目之间存在依赖关系（数据、交付、确认等需要跨项目流转）时，这些跨项目信息必须在会话边界之间持久化。仅靠聊天/口述的信息在 session 结束时丢失，接收方的下一个 agent 无从发现。Doc Harness 为这种场景提供一个**可选**的文件级协议。

本章是**自含的**——采用它的项目不需要任何本章和本项目 `DOC_HARNESS_SPEC.md` 之外的文档。无需外部协议文件，不依赖任何父目录或兄弟项目。不采用的项目完全不受影响。

### 14.1 何时采用

满足以下任一条件时采用：
- 本项目依赖其他项目（消费其数据、API 或交付物）。
- 其他项目依赖本项目。
- 跨项目协调反复发生（决定、需求、数字交换超过一次）。

真正独立的项目可以跳过。

### 14.2 目录结构

项目根目录下两个文件夹：

```
project_root/
├── inbox/        ← 发给本项目的消息
├── outbox/       ← 本项目发出的消息（底稿 / 已发送副本）
└── ... (项目其他内容)
```

初始为空。如果 VCS 需要（如 Git），可用 `.gitkeep` 让空目录被追踪。

### 14.3 消息格式

**文件名**：`YYYY-MM-DD-from-<source-project>-<topic-slug>.md`

- `<source-project>`：发送方项目短标识（如 `whoami`, `smss`）。
- `<topic-slug>`：主题的 kebab-case 概括。
- 示例：`2026-04-19-from-whoami-skill-improvement-proposals.md`。

**同日冲突消歧**：若文件名会与已存在文件冲突（相同日期 + 相同发送方 + 相同主题），在 `.md` 前追加 `-HHMMSS`（发送方本地时间，零填充）。示例：`2026-04-19-from-whoami-skill-improvement-proposals-143052.md`。发送方在写入前检查自己的 outbox 是否存在冲突；若有，在新消息上添加 HHMMSS，保留原件不动。接收方把带消歧后缀的文件名视作不同消息。

**亚秒级冲突**：罕见情况下（自动发送方每秒多条），即使 `-HHMMSS` 名也可能冲突。如加上 `-HHMMSS` 后仍有冲突，继续追加 `-<N>`——N 取能产生唯一名字的最小正整数（如 `...-143052-2.md`）。这是终止规则——不得退化为覆盖。§14.8"同名多份"行处理在上述规则之后仍发生冲突的病态情况（如两个发送方独立同时写入）：磁盘上的那份为准，通过 `/doc-harness check` §1.7 告知发送方。

**头部**（YAML frontmatter，必填）：

```yaml
---
from: <发送方项目名>
to: <接收方项目名>
date: YYYY-MM-DD
subject: <一行主题>
status: unread | read | actioned
in-reply-to: <原文件名，如为回复；否则 null>
priority: low | normal | high
---
```

**正文**：自由 Markdown。允许表格、清单、代码块。

### 14.3.1 发送前验证（发送方责任）

在向接收方 `inbox/` 写入新消息前，发送方**必须**先验证接收方已启用 inbox/outbox 协议，否则会造成静默失败：文件落入接收方从不扫描的目录，或因目标不存在而 `ENOENT` 失败。

验证步骤（按顺序；命中任一失败即停止）：
1. **目标路径存在**：接收方项目目录存在且发送方可访问。
2. **接收方 `inbox/` 存在**：若无，则接收方尚未采用协议（或已不再参与）。**不要替接收方创建 `inbox/` 目录**——那相当于未经许可地为其开启协议。
3. **接收方 CLAUDE.md 含有 inbox/outbox 铁律块**（对接收方 CLAUDE.md Grep 字符串 `inbox/`），这确认接收方的下一个 agent 确实会扫描 inbox。

**任何一步失败**：**不要**把文件复制到接收方 `inbox/`。改为：
- 按常规写入自己 `outbox/`（作为永久记录）。
- 在 CURRENT_STATUS 车身记录送达失败：`#### 尝试向 <接收方> 发送 <主题>（YYYY-MM-DD）——接收方未采用 inbox/outbox 协议，消息仅存于自己 outbox。`
- 下次与用户交互时提出。**不要**自动重试。

### 14.3.2 发送前自检清单（发送方自我保护）

在撰写跨项目消息前，发送方**必须**运行以下 5 项自检。这可以防止 agent 因身份混淆而误将其他项目当成自己的项目。

```markdown
- [ ] `from:` 字段写的是**本项目**的名称（不是接收方的）
- [ ] 消息写入的是**本项目**的 `outbox/`（不是接收方的）
- [ ] 复制目标是接收方的 `inbox/`（不是他们的 `outbox/`）
- [ ] **没有**修改接收方的任何内部文档
      （CURRENT_STATUS.md、FILE_INDEX.md、WORKLOG.md、代码、配置）
- [ ] 接收方已启用 inbox/outbox 协议（§14.3.1 验证已通过）
```

**任一项目失败**：停止。不要发送。先修正错误再继续。

### 14.4 消息生命周期与规则

```
发送方                                         接收方
  │                                              │
  ├─ 写入自己的 outbox/                           │
  ├─ 复制同一文件到接收方 inbox/                   │
  │  （发送方设置 status: unread）                │
  │                                              │
  │   ─── session 边界 ───                       │
  │                                              │
  │                      接收方的恢复链 ─────────┤
  │                      指向 inbox/             │
  │                      接收方读取消息 ─────────┤
  │                      → status: read          │
  │                      执行所需动作 ───────────┤
  │                      → status: actioned      │
  │                      （如需回复：            │
  │                       写一条新消息）         │
```

**规则**：

1. **收到的消息不可变**——永不修改 `inbox/` 内的文件，仅更新 `status` 字段。需回复就写新消息。
2. **status 转换**——发送方投递时设为 `unread`。接收方读后改为 `read`，执行完所需动作后改为 `actioned`。无需动作的消息直接 `actioned` 即可视为已确认关闭。
3. **归档**——`status: actioned` 超过 30 天的消息**应**迁入 `inbox/_archive/`（内部结构相同）。归档审计由 `/doc-harness check` §1.7 执行——当检查发现 ≥5 条超过 30 天的 actioned 消息时，会把归档作为行动项提出；agent 在本 session 内执行迁移。归档消息是永久记录，不得删除。
4. **发送方保留底稿**——发送方的 `outbox/` 是永久记录。若接收方丢失副本，以发送方 outbox 为权威。
5. **快照优于指针**——向其他项目通报数字、交付、状态（"我们模型精度达到 0.88"）时，把值直接写在消息正文里，而**不要**让对方指向你的内部文件。内部文件会随重构漂移，快照一旦投递不会说谎。

### 14.5 与四核心文档的集成

**CLAUDE.md** —— 添加一个铁律块（可直接复制粘贴，无需定制）：

```markdown
**项目间通讯走 inbox/outbox 文件协议**

- **机制**：本项目维护 `inbox/`（收信）和 `outbox/`（发信）。消息是 YAML frontmatter + Markdown 的 `.md` 文件，文件名格式 `YYYY-MM-DD-from-<source>-<topic>.md`。
- **为何强制（如已采用）**：跨项目信息仅在聊天中传递会在 session 结束时丢失。文件级消息跨越 session 边界保存，并通过本项目的恢复链被任何未来 agent 发现。
- **接收**：恢复链检查 `inbox/` 中 `status: unread` 的消息。读取 → 改为 `status: read`。执行动作后 → `status: actioned`。永不修改收到的消息（仅 status 字段），需回复则写新消息。
- **发送**：写入本项目的 `outbox/`（发送方永久底稿）并复制同一文件到接收方的 `inbox/`。在 CURRENT_STATUS 中记录该次通讯以便下次会话可追溯。
- **快照优于指针**：向其他项目通报数字/交付时，把值写在消息正文里，不要让对方引用你的内部文件。
- **完整规范**：`DOC_HARNESS_SPEC.md` §14（本项目根目录）。
```

**CURRENT_STATUS.md** —— 需要动作的入信加入车灯：

```markdown
### 立即行动
- [ ] 回复 <发送方> 的 <主题>（见 `inbox/YYYY-MM-DD-from-<source>-<topic>.md`）
```

发出的通讯随手记入车身：

```markdown
#### 向 <接收方> 发送交付物（YYYY-MM-DD）
- 消息：`outbox/YYYY-MM-DD-from-<本项目>-<主题>.md`（与 §14.3 文件名规范一致——outbox 中的文件与发往收件方 inbox 的副本同名）
- 概要：<一行>
```

**FILE_INDEX.md** —— 两个目录单列类别：

```markdown
## 项目间通讯
- `inbox/` — 来自其他项目的消息
- `outbox/` — 发出的消息（底稿与已发副本）
```

**恢复链（CLAUDE.md）** —— 任务条件读加一条：

```markdown
### 任务条件读
- 如 `inbox/` 有任何 `status: unread` 的文件：优先处理
- ...
```

注意：这条在**任务条件读**层，不在必读。每次 session 开始把整个 inbox 读完会烧 context；只有 unread 消息要紧。inbox 为空或都是 read/actioned 时这条自然为 no-op。

### 14.6 给"首次遇到本机制"的 agent

如果你进入一个项目，在根目录看到 `inbox/` 和 `outbox/`，说明该项目使用本协议。请这样做：

1. 扫描 `inbox/`，找所有 frontmatter 中 `status: unread` 的文件（frontmatter schema 与生命周期词汇见 §14.3）。
2. 逐一处理：读取、执行所需动作（如有）、更新 `status` 字段（读后 `read`，完成后 `actioned`）。
3. 在修改本项目内部文件（可能被其他项目引用的模型、接口、数字）之前，考虑是否需要向依赖方 inbox 发一份快照。
4. 发送方法：写入本项目 `outbox/` 并复制同一文件到接收方 `inbox/`（同文件名——见 §14.3）。在 CURRENT_STATUS 中记录。
5. 如遇到看起来异常的 inbox 文件（无 frontmatter、未知 status 值、文件名不符规范），按 §14.8 处理，不要猜测。

你需要的一切都在本章。无需外部文件。

### 14.7 在已有项目中追溯启用

现有项目后续启用的步骤：
1. 在项目根创建 `inbox/` 和 `outbox/`。
2. 从 §14.5 复制铁律块到 CLAUDE.md 的铁律章节。
3. 把任务条件读条目加到恢复链（§14.5 中）。
4. 在 FILE_INDEX 加 `## 项目间通讯` 类别。
5. （可选）向已知依赖方项目发一条通知消息，告知本项目已可通过 inbox 联系。

### 14.8 异常消息处理

当 `inbox/` 中出现不符合 §14.3 的文件时，**不要静默跳过**、**不要猜测**。按下表分类，一次性处理——不对异常文件进入重试循环。

| 症状 | 宽松还是严格？ | 处理 |
|------|--------------|------|
| 文件名不符 `YYYY-MM-DD-from-<source>-<topic>.md`，但 YAML 有效 | 宽松 | 按正常流程处理。在 CURRENT_STATUS 车身记录文件名异常。**不重命名文件**（它是发送方的记录，按 §14.4 规则 1 不可变）。 |
| 完全无 YAML frontmatter | 严格 | 移入 `inbox/_malformed/`（如不存在则创建）。在 CURRENT_STATUS 车身写一行：`#### 隔离异常消息 inbox/<filename>（无 YAML），YYYY-MM-DD`。**不尝试**解析正文。 |
| YAML frontmatter 存在但缺 `status:` 字段 | 严格 | 为安全起见按 `unread` 处理（偏向多做动作）。补上 `status: unread` 再读。在车身记录异常。 |
| `status:` 是未知值（非 `unread`/`read`/`actioned`） | 严格 | 按 `unread` 处理。在车身记录异常。读后设 `read`（必要时再 `actioned`），使用规范值。 |
| 必填 frontmatter 字段缺失（`from`、`to`、`date`、`subject`） | 严格 | 移入 `inbox/_malformed/`，在车身写一行记录。**不伪造**缺失值。四个字段均为 §14.3 必填。 |
| `priority:` 缺失或取值未知 | 宽松 | 按 `priority: normal` 处理，在车身记录异常。 |
| `in-reply-to:` 完全缺失 | 宽松 | 按"非回复"处理（null）。无需动作。 |
| `in-reply-to:` 指向的本项目 outbox 文件不存在 | 宽松 | 按正常流程处理；在车身记录悬空引用，但不阻塞。 |
| 同名文件多份（§14.3 消歧规则应用后仍偶发，如两个发送方独立同时写入） | 严格 | 磁盘上的那份为准。在车身记录：`#### 观察到文件名冲突 <名>——保留磁盘副本`。**不合并**内容。通过 `/doc-harness check` §1.7 告知发送方，以便必要时用不同 topic-slug 重发。 |

`inbox/_malformed/` 不是归档——它是隔离区，供 agent / 用户检查。该目录的条目需人工审查，不得被静默遗忘。`/doc-harness check` §1.7 会报告数量。

## 附录E：设计选择（FAQ）

### 为什么 inbox/outbox 是可选机制，不是强制的？

因为很多项目是独立的，没有依赖方。强制每个项目都装载 inbox/outbox 会产生空目录和用不上的铁律，削弱对 skill 的信任。机制作为**标准可选特性**提供，附带完整自含规范（第十四章），需要的项目可一致采用，不需要的项目忽略即可。

### 为什么机制写在 doc-harness 内部，而不是独立 spec？

因为外部 spec 是**脆弱的**。项目内部工作的 agent 对外部文件没有可靠的入口；长期项目演化中，外部引用容易丢失、被误解、被遗忘。把完整机制嵌入 doc-harness spec 之后，每个部署了 `DOC_HARNESS_SPEC.md` 的项目都自带一份自含描述——项目自身成为"该协议在本项目里如何运作"的权威来源。

### 为什么 PARKING_LOT.md 和 PHILOSOPHY.md 是可选的？

不是每个项目都有值得记录的暂缓事项或跨项目原则。强制要求会产生空壳文件，削弱对四文档体系的信任。把它们作为模式提供，让每个项目在真的有内容时再采用。

### 为什么恢复链必读层保持在 2–3 条？

再大，"必读"就会变成"大概应该读"，agent 要么开始跳过条目，要么为保险起见读完整个项目。两层切分只有在基线真的精简时才起作用。只对*部分*工作相关的条目应放到任务条件读。

### 为什么用行数（而非阶段数）作为 WORKLOG 归档触发条件？

阶段密度在项目间差异巨大——从基础设施工作的每阶段 ~20 行，到密集研究工作的每阶段 ~200 行。行数反映读取此文件的实际成本，这才是我们关心的。
