# 状态文档体系规范（Status Documentation System Specification）

**版本**: v1.3
**日期**: 2026-04-19
**状态**: 正式版，可用于项目实施

---

## 第一章 首要原则与概述

### 1.1 首要原则："落笔为安"（Write It Down or Lose It）

Context中的一切信息都是临时的。无论多重要的分析结果、多精妙的设计思路，只要没有**写入文件**并**注册到FILE_INDEX**，就等于不存在。

- **落笔**：重要信息必须写入文件。不写 = 丢失。
- **为安**：写入的文件必须注册到FILE_INDEX。不注册 = 虽存在但不可发现 = 等同于丢失。

这条原则优先于一切效率考量。宁可多写一个文件，不可让信息只存在于context中。

### 1.2 什么是状态文档体系

状态文档体系是一个由四个核心文档组成的信息管理系统，服务于一个**项目**。

| 文档 | 角色 | 核心职责 | 更新频率 |
|------|------|---------|---------|
| **CLAUDE.md** | 门牌 | 项目入口、恢复起点、项目级铁律 | 阶段切换时 |
| **CURRENT_STATUS.md** | 行驶中的汽车 | 当前阶段活跃信息、近期历史摘要、下一步、阶段级原则 | 每个session |
| **FILE_INDEX.md** | 图书馆目录 | 按类别索引项目所有文件 | 文件变化时 |
| **WORKLOG.md** | 行车记录仪 | 永久保存所有阶段的完整工作详情 | 阶段切换时（批量） |

### 1.3 核心场景

| 场景 | 需求 |
|------|------|
| **全新Agent到达** | 读CLAUDE.md → CURRENT_STATUS → 知道做什么 |
| **Context compact后恢复** | 读CURRENT_STATUS → 从车身恢复到中断点 |
| **跨项目切换** | 读目标项目CLAUDE.md → 快速了解 |
| **工作过程中** | 实时更新CURRENT_STATUS车身 + 新文件注册FILE_INDEX |

### 1.4 不依赖外部记忆系统

状态文档体系完全自足。不依赖MEMORY.md、不依赖对话历史、不依赖任何Agent的私有记忆。任何Agent仅凭阅读文件即可完全恢复工作状态。

如果环境提供了记忆系统（如Claude Code的MEMORY.md），可以作为**补充加速**（提供额外上下文），但不作为恢复的必要条件。

## 第二章 CLAUDE.md — 入口文档

### 2.1 职责
- 告诉来者"这是什么项目"
- 提供恢复链（阅读顺序）
- 定义项目级铁律（永久有效，不随阶段变化）
- 指向其他三个核心文档
- 提供项目特有的稳定技术信息

### 2.2 不做什么
- 不记录详细当前状态（→ CURRENT_STATUS）
- 不罗列所有文件（→ FILE_INDEX）
- 不记录历史工作详情（→ WORKLOG）

### 2.3 模板

CLAUDE.md由三部分组成：项目特定信息 + 操作版规则（嵌入） + 完整版规范指针。

```markdown
# [项目名] — 入口文档

**最后更新**: YYYY-MM-DD
**当前阶段**: [阶段名称]（⏳/⏸️/✅）
**一句话状态（截至YYYY-MM-DD）**: [简述]

---

## 恢复链

恢复链是任何 agent 或人类继续本项目工作的入门仪式，分两层。

### 必读（基线）

任何继续工作的人都必须读的文件。保持在 2–3 条。

1. 本文件（CLAUDE.md）
2. `CURRENT_STATUS.md`

### 任务条件读

只在工作匹配时读。随项目演化增减。

- 如要查找具体文件：读 `FILE_INDEX.md`
- 如要查历史阶段详情：读 `WORKLOG.md`
- [按需添加项目特定条目——如"做 X 时读 Y.md"]

### 元规则

- 恢复链必须**自含**：条目只指向项目内（或兄弟项目稳定路径的）文件。不依赖 agent 侧特性（memory 系统、对话历史、外部服务）——下一个 agent 可能没有这些。
- 恢复链是**活的**：有新类型工作出现时加条件读条目；旧类别退役时删除。阶段切换时审查。

## 项目概述

[3-5行：项目目标、产出、关键背景]

## 项目级铁律

[跨阶段永久有效的不可违反规则]
- 铁律1：...
- 铁律2：...

## 关键技术信息

[稳定的技术细节：编译命令、路径约定、依赖关系等]

## 子项目（如有）

- `子项目名/` → 见 [子项目/CLAUDE.md](子项目/CLAUDE.md)（状态：⏳/⏸️/✅）

## 状态文档体系——操作规则

[此处嵌入 OPERATIONAL_RULES.md 的完整内容（约106行）。
 操作规则是Agent日常维护状态文档的完整指南。
 完整设计原理见项目根目录的 STATUS_DOC_SYSTEM_SPEC.md。]
```

操作规则的完整内容见本项目 `OPERATIONAL_RULES.md`。`/status-init` skill在创建CLAUDE.md时会自动嵌入当前版本的操作规则。

### 2.4 更新规则
- 阶段切换时更新"当前阶段"和正文内容
- "一句话状态（截至日期）"每session结束时刷新
- 新增铁律时更新"项目级铁律"
- 操作规则部分一般不修改——它由创建时的规范版本决定

## 第三章 CURRENT_STATUS.md — 当前状态文档

### 3.1 职责与结构

CURRENT_STATUS是项目的**活跃神经中枢**，使用"行驶汽车"比喻组织内容：

| 区域 | 比喻 | 内容 | 生命周期 |
|------|------|------|---------|
| §1 近期已完成 | 车辙 | 最近2-3个已完成阶段的简要总结（各3-5行） | 滚动窗口 |
| §2 当前工作 | 车身 | 当前阶段的全部详细工作记录 | 阶段结束时→WORKLOG |
| §3 下一步 | 车灯 | 立即行动 + 后续规划 | 每session更新 |
| §4 当前工作原则 | 驾驶手册 | 本阶段适用的注意事项和原则 | 随阶段生灭 |

### 3.2 模板

```markdown
# CURRENT_STATUS — [项目名]

**最后更新**: YYYY-MM-DD
**当前阶段**: [阶段名称]

---

## 近期已完成（车辙）

（项目刚启动时写"暂无已完成阶段"）

### [阶段N-1名称]（YYYY-MM-DD ~ YYYY-MM-DD）
[3-5行摘要]
详细记录见 WORKLOG.md § [对应标题]。

### [阶段N-2名称]（YYYY-MM-DD ~ YYYY-MM-DD）
[3-5行摘要]
详细记录见 WORKLOG.md § [对应标题]。

---

## 当前工作（车身）

### 阶段目标
[本阶段要完成什么]

### 已完成步骤

#### [步骤1名称]（YYYY-MM-DD）
- 做了什么
- 创建了什么文件 → 已注册FILE_INDEX
- 关键发现/决策

#### [步骤2名称]（YYYY-MM-DD）
- ...

### 待解决问题
- [问题1]

---

## 下一步（车灯）

### 立即行动
1. [下一个具体步骤]

### 后续规划
- [更远的计划]

---

## 当前工作原则（驾驶手册）

[本阶段适用的注意事项。阶段切换时逐条审查：(a)升级为铁律？(b)保留？(c)移除？
 原则在阶段内被满足时标注[已满足]但保留到阶段结束。]
- 原则1：...
- 原则2：...
```

### 3.3 车身的更新粒度

**关键规则：为compact做好准备。** 不要假设当前session能完整结束。

- 每完成一个**有意义的步骤**后，更新车身。"有意义"= 如果此刻丢失了context，下一个Agent需要知道这一步已完成。
- 车身内按**步骤**（非session）组织。一个session可能完成多个步骤，一个步骤也可能跨多个session。
- **车身长度建议**：控制在约200行以内。如果超过，考虑是否应触发阶段切换。

### 3.4 更新规则
- 每个session至少更新一次（车身的已完成步骤 + 车灯的下一步 + 最后更新日期）
- 创建新文件时，在车身中记录"创建了xxx"，同时注册FILE_INDEX

## 第四章 FILE_INDEX.md — 文件索引文档

### 4.1 职责
按**类别**索引项目所有文件。回答"项目里有哪些文件、是什么、在哪里"。

### 4.2 组织原则
- 按类别（##标题）分组，**不按时间排列**
- 类别是项目特定的，可以随时新增或重组
- 每个条目：文件路径 + 一句话说明
- 如果一个文件横跨两类，只放在主类别中，另一类别下用"→ 另见 §[主类别]"指向

### 4.3 递归子索引

当某类别的文件数量超过**10个**时，建议在该类别文件夹中创建子FILE_INDEX.md。超过**20个**时，必须创建。

父FILE_INDEX对该类别只保留一行：
```markdown
- `notes/findings/` → 见 [`notes/findings/FILE_INDEX.md`](notes/findings/FILE_INDEX.md)
  （Ch04/Ch05研究发现文档，共38个，按章节分类）
```

子FILE_INDEX头部标注父索引：
```markdown
# FILE_INDEX — [类别名]
**父索引**: [`../../FILE_INDEX.md`](../../FILE_INDEX.md)
```

递归深度限制为**2层**（父→子→孙）。如果需要更深，说明项目结构过于复杂，应考虑重组文件夹。

### 4.4 批量文件创建

当一个session创建大量同类文件（如一次性生成20个分析报告），不必逐一注册。可以：
1. 在FILE_INDEX中注册**文件夹**，标注文件数量和命名规则
2. 如果文件数量>10，创建子FILE_INDEX.md
3. 在CURRENT_STATUS车身中记录"创建了N个文件于xxx/，命名规则为..."

### 4.5 模板

```markdown
# FILE_INDEX — [项目名]文档目录

**最后更新**: YYYY-MM-DD

---

## 核心状态文档
- `CLAUDE.md` — 项目入口
- `CURRENT_STATUS.md` — 当前状态
- `FILE_INDEX.md` — 本文件
- `WORKLOG.md` — 工作历史

## [类别1名称]
- `path/file1.md` — 说明
- `path/file2.md` — 说明

## [类别2名称]
- `path/folder/` → 见 [`path/folder/FILE_INDEX.md`]（N个文件，简述）

## [类别3名称]
- `path/file3.py` — 说明
```

### 4.6 更新规则
- 创建新文件时注册
- 删除/归档文件时移除
- 不需要每个session更新——只在文件系统发生变化时

## 第五章 WORKLOG.md — 工作日志文档

### 5.1 职责
永久保存所有阶段的**完整**工作详情。内容来源于CURRENT_STATUS阶段切换时的转移。

### 5.2 头部目录（TOC）

WORKLOG在文件顶部维护一个目录，方便在长文档中快速定位：

```markdown
# WORKLOG — [项目名]

## 目录
| 阶段 | 时间 | 页内锚点 |
|------|------|---------|
| 阶段5: 格式修正 | 2026-03-29~03-30 | [→](#阶段5-格式修正) |
| 阶段4: 文献补强 | 2026-03-27~03-29 | [→](#阶段4-文献补强) |
| 阶段3: Ch5重建 | 2026-03-25~03-27 | [→](#阶段3-ch5重建) |
| ... | | |

---

## 阶段5: 格式修正
[详细内容...]

## 阶段4: 文献补强
[详细内容...]
```

### 5.3 内容格式

每个阶段是一个顶级章节（##），包含：

```markdown
## 阶段N: [名称]（YYYY-MM-DD ~ YYYY-MM-DD）

### 阶段总结
[阶段结束时添加的3-5行总结]

### 详细记录
[从CURRENT_STATUS车身区原封不动复制的内容]
```

### 5.4 修改规则

- **允许**：在已有阶段的开头**追加**总结或勘误注释
- **允许**：在顶部目录中添加新条目
- **禁止**：修改或删除从CURRENT_STATUS复制来的原始内容
- 事实性错误以勘误方式标注：`[勘误(YYYY-MM-DD): 实际为4087篇，非4090篇]`

### 5.5 归档

WORKLOG 只增不改，必然增长。一个 2000 行的 WORKLOG 本身就是负担——任何被要求"查历史"的 agent 一次取走全部就烧光了 context。归档就是让 WORKLOG.md 本身保持可读，同时在磁盘上完整保留历史。

**触发**（软阈值）：WORKLOG.md 超过**约 1000 行**。用行数而非阶段数衡量——各项目阶段密度相差 10 倍，行数才是可靠单位。

**归档流程**：

1. 决定切点：`WORKLOG.md` 保留最近 **3 个阶段**，更早的迁出。
2. 创建 `WORKLOG_ARCHIVE_<YYYY-QN>.md`（按季度粒度，如 `WORKLOG_ARCHIVE_2026-Q1.md`）。用被归档阶段的时间范围确定季度标签。归档文件可按时间累积多份。
3. 将更早的阶段章节（含目录表条目）迁入归档文件。归档文件结构与 WORKLOG.md 相同（顶部目录 + 阶段章节）。
4. 在 `WORKLOG.md` 标题下方紧接着添加一行指针：
   ```
   > 更早的历史归档于：WORKLOG_ARCHIVE_2026-Q1.md, WORKLOG_ARCHIVE_2026-Q2.md, ...
   ```
5. 在 FILE_INDEX 的 `## 历史归档` 类别下注册每个归档文件（如类别缺失则创建）。

**非目标**：归档不是删除。归档文件是永久记录，除勘误外不再修改，永不删除。

**何时可跳过**：阶段密度小（如每阶段 ~20 行）的项目可能永远不触发。这没问题——阈值只捕捉确实需要的项目。

## 第六章 信息流动规则

### 6.1 日常工作流

```
读CLAUDE.md → 读CURRENT_STATUS → 开始工作
工作中：
  ├── 每完成一个有意义的步骤 → 更新CURRENT_STATUS车身
  ├── 创建新文件 → 注册FILE_INDEX + 记录在CURRENT_STATUS车身
  └── session结束 → 确认车身反映了全部工作 + 更新车灯 + 更新日期
```

### 6.2 阶段切换流（五步操作）

**触发条件**（满足任一即可触发）：
- 工作目标发生质变（如从"写作"转入"审读"）
- 用户明确宣布进入新阶段
- 项目暂停（超过3天无工作）
- 车身已超过200行

**不触发**：同一目标内的子任务切换（如文献补强中从Ch4到Ch5）

**五步操作**（按此顺序执行）：

```
Step 1（保护性操作，最先执行）:
  将CURRENT_STATUS车身全部内容复制到WORKLOG顶部（目录之后），
  添加阶段总结。
  → 此步完成后，即使后续步骤被中断，也不会丢数据。

Step 2:
  在CURRENT_STATUS车辙区最顶部插入新摘要（3-5行总结本阶段）。
  标注"详细记录见WORKLOG.md §阶段N"。

Step 3:
  如果车辙已有3个摘要，删除最老的。

Step 4:
  清空车身。填入新阶段的目标。
  更新驾驶手册：
    - 移除不再适用的原则
    - 检查是否有原则应提升为CLAUDE.md铁律
    - 添加新阶段适用的原则

Step 5:
  更新CLAUDE.md的"当前阶段"和"一句话状态（截至日期）"。
  更新WORKLOG的目录表。
```

**中断安全**：Step 1最先执行（将内容复制到WORKLOG），确保即使compact在Step 2-5之间发生，数据已安全保存。下一个Agent可以在WORKLOG中找到完整内容，并完成剩余步骤。

### 6.3 Context compact恢复流

```
Compact发生
  → 读CLAUDE.md → 了解项目和恢复链
  → 读CURRENT_STATUS.md：
      车辙 → "之前做完了什么"
      车身 → "当前做到哪里了"（如果车身有内容但似乎不完整，
             检查WORKLOG最新条目是否有更完整的记录——
             可能是阶段切换被中断）
      车灯 → "接下来做什么"
      驾驶手册 → "工作中注意什么"
  → 继续工作（如果用户在场，先确认下一步是否有变）
```

### 6.4 项目暂停与恢复

**暂停时**：
1. 执行阶段切换五步操作（将当前工作存入WORKLOG）
2. 在CURRENT_STATUS车身写："项目暂停于YYYY-MM-DD。暂停原因：..."
3. 在CLAUDE.md更新状态为⏸️

**恢复时**：
1. Agent读CLAUDE.md → 看到⏸️
2. 读CURRENT_STATUS → 车辙中有暂停前最后阶段的摘要，车身中有暂停说明
3. 读WORKLOG最新阶段 → 获取暂停前的完整工作详情
4. **关键**：车灯中的"下一步"可能已过期（暂停期间情况可能变化），应与用户确认后再执行

## 第七章 单一事实来源

### 7.1 核心规则

**每个事实只在一个文档中维护。**

| 事实类型 | 唯一来源 |
|---------|---------|
| 当前状态/进度/数字 | CURRENT_STATUS.md |
| 文件列表/位置 | FILE_INDEX.md |
| 历史工作详情 | WORKLOG.md |
| 项目概述/入口/铁律 | CLAUDE.md |

### 7.2 唯一允许的冗余

CLAUDE.md的"一句话状态（截至日期）"。必须带日期标记，矛盾时以CURRENT_STATUS为准。

## 第八章 工作原则的两级管理

### 8.1 两类原则

| 类型 | 存放位置 | 生命周期 | 示例 |
|------|---------|---------|------|
| **项目级铁律** | CLAUDE.md"铁律"区 | 与项目同寿 | "不得直接读取论文.md原文" |
| **阶段级原则** | CURRENT_STATUS"驾驶手册"区 | 随阶段生灭 | "本阶段每次修改LaTeX后必须编译验证" |

### 8.2 原则的升降级

- 阶段切换时，检查驾驶手册中是否有原则**应提升为项目级铁律**
- 如果一条原则在连续3个阶段都出现，大概率应该升级
- 升级：从驾驶手册移到CLAUDE.md铁律区
- 降级很少发生——铁律一旦确立通常不撤销

### 8.3 铁律的管理

铁律存放在CLAUDE.md中，与项目同寿。可以在阶段切换时新增铁律（从驾驶手册中提升）。铁律一般不删除——如果某条铁律的前提条件不再适用，标注"[已不适用: 原因]"但保留原文，以供历史参考。

## 第九章 文档生命周期

### 9.1 核心状态文档
四个核心文档与项目同寿。

### 9.2 叶子文档

| 状态 | 含义 | 标记方式 |
|------|------|---------|
| 活跃 | 正在使用或仍有效 | 默认（不标记）|
| SUPERSEDED（保留） | 已被替代，但仍有参考价值 | 文件首行：`⚠️ SUPERSEDED BY [新文档]`。在FILE_INDEX中移至 `## 已替代` 类别。|
| SUPERSEDED（归档） | 已被替代且不再需要 | 文件首行：`⚠️ SUPERSEDED BY [新文档]`。移入_archive/，从FILE_INDEX移除。|
| ARCHIVED | 历史文档，已移入_archive/ | 从FILE_INDEX移除 |

实践中，大多数被替代的文档应当**保留**而非归档，因为它们往往包含有价值的参考信息（如原始README中的风格指南、旧STATUS中的详细session记录）。除非文档确实已无任何残余价值，否则优先使用"保留"方式。

## 第十章 子项目与项目群

### 10.1 子项目
子项目拥有完整的四文档体系，递归适用本规范。父项目FILE_INDEX中注册子项目入口。

### 10.2 项目群
多个独立项目共享父目录时，父目录可有一个轻量级CLAUDE.md（仅导航表，无CURRENT_STATUS/FILE_INDEX/WORKLOG）。

```markdown
# [工作区] — 项目导航

## 活跃项目
| 项目 | 入口 | 状态 |
|------|------|------|
| 项目A | [CLAUDE.md](A/CLAUDE.md) | ⏳ |
| 项目B | [CLAUDE.md](B/CLAUDE.md) | ⏸️ |

## 共享基础设施
| 工具 | 路径 | 说明 |
|------|------|------|

## 归档项目
| 项目 | 路径 | 完成日期 |
|------|------|---------|
```

Agent到达项目群时，通常由用户指令决定进入哪个子项目。

## 第10.3节 适应已有项目

**核心原则：状态文档体系适应项目，而非项目适应体系。**

### 中途引入

现实中大多数引入发生在项目进行过程中——项目已有文件、可能已有状态文档、也有一定的工作历史。引入流程如下：

1. **先理解项目。** 在创建状态文档之前，仔细阅读所有现有的状态/说明文档。了解项目目标、当前状态、目录结构以及已有的文档。

2. **创建四个核心文档。** CLAUDE.md、CURRENT_STATUS.md、FILE_INDEX.md、WORKLOG.md——填入项目的实际当前状态，而非空白模板。

3. **处理已有的状态文档。** 大多数应标记为 SUPERSEDED 但保留：
   - 在文件首行添加 `⚠️ SUPERSEDED BY [新文档]`
   - 在 FILE_INDEX 中移至 `## 已替代` 类别
   - 注明保留原因（如"包含详细的session记录"或"含有风格指南可供参考"）

4. **从历史记录重建 WORKLOG。** 利用现有日志、状态文档和git历史来构建阶段记录。重建的部分应标注："（根据[来源文档]重建）"。目标是有用的历史，而非完美的精确度——实时记录的细节度始终更高。

5. **保留已有的高质量索引。** 如果某个子目录已有维护良好的索引（如含100+条目和状态标签的研究分析INDEX.md），FILE_INDEX.md应当**指向它**而非重建。添加一行即可：`→ 见 [subdir/INDEX.md]（N个条目，覆盖[范围]）`。

### 大型目录的子索引指引

对于超过20个文件的目录，创建子FILE_INDEX.md。初次创建大型子索引（100+文件）时，可通过文件系统扫描自动生成初始结构，然后在可能的情况下进行人工验证和标注。

## 第十一章 Agent行为规范

### 11.1 Session开始——应用恢复链

按项目 CLAUDE.md 中定义的恢复链执行：

1. 按顺序读**必读**层（CLAUDE.md → CURRENT_STATUS.md，通常 2–3 个文件）。
2. 扫描**任务条件读**层。只读条件匹配当前工作的条目。
3. 如果用户在场 → 确认"下一步"是否仍然正确。
4. 如果是 compact 恢复（用户不在场）→ 直接按车灯继续。

不要在启动时无条件读完所有任务条件读条目——那会让两层切分失去意义。如果你发现某条条件读条目每次都读，它其实属于必读层，应显式提升。

### 11.2 工作过程中

1. 每完成一个**有意义的步骤** → 更新CURRENT_STATUS车身
2. 创建新文件 → **同时做两件事**：(1)注册FILE_INDEX (2)记录在CURRENT_STATUS车身
3. **"落笔为安"**：重要分析结果、设计思路、决策理由 → 写入文件 + 注册
4. **关注剩余 context**（如运行环境暴露该信息）：context 压缩等同于非自愿的 session 结束。如果环境报告 context 使用率（百分比或 token 数），将**剩余不足**（~<20%）视为立即触发——在下一次工具调用前更新 CURRENT_STATUS；如果车身已积累大量未保存的工作，立即考虑阶段切换。别等那个可能永远不会到来的"有意义的步骤"。

### 11.3 Session结束

1. 确保CURRENT_STATUS车身反映了本session的全部工作
2. 更新车灯的"下一步"
3. 更新CURRENT_STATUS的"最后更新"日期
4. 更新CLAUDE.md的"一句话状态（截至日期）"
5. 检查：本session创建的文件是否都注册了FILE_INDEX
6. 检查：有没有重要信息只存在于context中？→ 落笔为安
7. 如发生了阶段切换：检查 WORKLOG 长度——超过约 1000 行？→ 按 §5.5 触发归档。

### 11.4 阶段切换

执行第六章§6.2的五步操作。

## 第十二章 反模式

| 反模式 | 正确做法 |
|--------|---------|
| 同一数字在多个文档中硬编码 | 只在CURRENT_STATUS中维护，其他文档引用 |
| 被替代的文档不标注 | 首行添加SUPERSEDED标记 |
| 多条恢复链 | 恢复链只在CLAUDE.md中定义 |
| 恢复链依赖 agent 侧特性（memory / 对话历史 / 外部服务）| 恢复链必须自含——只指向项目内文件 |
| 恢复链必读层超过 3 条 | 多出的降级到任务条件读；必读层保持精简 |
| 创建文件不注册 | 创建即注册FILE_INDEX |
| CURRENT_STATUS无限膨胀 | 阶段切换时详情转移到WORKLOG |
| WORKLOG 超过约 1000 行未处理 | 按 §5.5 归档 |
| 信息只存在于context中 | "落笔为安"——写入文件并注册 |
| 阶段级原则变成永久遗产 | 阶段切换时审查：升级为铁律或移除 |
| WORKLOG无目录 | 头部维护TOC表 |

## 第十三章 可选长期文档

核心四文档（CLAUDE / CURRENT_STATUS / FILE_INDEX / WORKLOG）面向中短期工作流：当前阶段是什么、做完了什么、文件在哪、历史怎样。有两类信息不适合放进这四份里。Doc Harness 为它们提供两个**可选**文档，只在项目真正积累相应内容时才使用。

这些文档是可选项。不需要时就不要创建。

### 13.1 PARKING_LOT.md — 暂缓事项

**目的**：想做但现在做不了的具体事。没有它，这类事项要么挤在 CURRENT_STATUS 车灯里（错，它们不是"下一步"），要么被新工作覆盖而遗忘。

**何时创建**：当某个具体事项正从当前视野移出、但又不应丢失。典型触发：阶段切换时，某条驾驶手册原则或下一步事项不再属于*当下*，但在已知条件下未来会重新相关。

**条目格式**：每条记录**要做什么**、**为何推迟**、**复活的前置条件**、**何时复查**。

```markdown
## 条目

### [事项标题] — 推迟于 YYYY-MM-DD
- **要做**：[具体行动或目标]
- **为何推迟**：[原因——如"依赖 X，X 要到 Q3 才发布"]
- **复活前置条件**：[什么为真时可以恢复]
- **复查**：[日期或事件——如"下次阶段切换时复查"或"2026-09-01"]
```

**生命周期**：前置条件满足时，要么 (a) 将事项迁回 CURRENT_STATUS 车灯，从 PARKING_LOT 划除；要么 (b) 保留条目并标注决定——如"[已复活 2026-09-15 → 现为活跃]"或"[已不再相关 2026-09-15 因为 X]"。不得悄悄删除。

**与恢复链的接口**：加入任务条件读层，如 `- 规划新阶段时：读 PARKING_LOT.md（有无可复活的暂缓事项？）`。

### 13.2 PHILOSOPHY.md — 来自实践的原则

**目的**：记录本项目具体实践催生的、可推广的原则、教训、洞察。这类内容比铁律更宽（铁律管*本项目*的操作），比驾驶手册更久（驾驶手册管*本阶段*）。

**设计原理**：通用的智慧来自具体的实践。项目是炉子，原则是从中出来的东西。禁止项目记录自己的哲学，等于切断原则与其诞生地的联系。PHILOSOPHY.md 就是这个捕获层。

**何时创建**：在项目工作中浮现一条洞察、你怀疑它可推广超出当下情境——一条可复用的教训、一个值得记住的设计权衡、一个不想重蹈的错误模式。

**条目格式**：每条陈述原则、催生它的实践、首次记录日期。

```markdown
## 原则

### [简短原则名] — 首次记录 YYYY-MM-DD
**原则**：[一句话陈述]

**源于**：[催生它的具体实践/事件/观察——具体说出*发生了什么*导致这条教训]

**范围**：[适用在哪——仅本项目？可能更广？已确认更广？]
```

**跨项目共享（可选）**：当一条原则被证明适用于本项目之外，可复制到兄弟项目自己的 `PHILOSOPHY.md`，或（若项目群存在一个父级导航 CLAUDE.md——见第十章）在那里引用。复制/引用**不移除**本项目 `PHILOSOPHY.md` 中的条目——每个采用该原则的项目各自保留诞生记录，本文件记录*本项目*如何得出这条原则。共享的条目可标注 `[亦被 <项目> 采用 YYYY-MM-DD]` 以便追溯。

**生命周期**：原则可以勘误精修（`[精修 YYYY-MM-DD: ...]`）或标注被替代（`[YYYY-MM-DD 被替代，新表述为: ...]`），但条目不删除。PHILOSOPHY.md 读得少，但应完整保留。

**与恢复链的接口**：用于反思性决策，加入任务条件读层，如 `- 做重大架构决策时：读 PHILOSOPHY.md`。

## 第十四章 可选的跨项目通信（inbox/outbox）

当一个项目与其他项目之间存在依赖关系（数据、交付、确认等需要跨项目流转）时，这些跨项目信息必须在会话边界之间持久化。仅靠聊天/口述的信息在 session 结束时丢失，接收方的下一个 agent 无从发现。Doc Harness 为这种场景提供一个**可选**的文件级协议。

本章是**自含的**——采用它的项目不需要任何本章和本项目 `DOC_HARNESS_SPEC.md` 之外的文档。无需外部协议文件，不依赖任何父目录或兄弟项目。不采用的项目完全不受影响。

### 14.1 何时采用

满足以下任一条件时采用：
- 本项目依赖其他项目（消费其数据、API 或交付物）。
- 其他项目依赖本项目。
- 跨项目协调反复发生（决定、需求、数字交换超过一次）。

真正独立的项目可以跳过。

### 14.2 目录结构

项目根目录下两个文件夹：

```
project_root/
├── inbox/        ← 发给本项目的消息
├── outbox/       ← 本项目发出的消息（底稿 / 已发送副本）
└── ... (项目其他内容)
```

初始为空。如果 VCS 需要（如 Git），可用 `.gitkeep` 让空目录被追踪。

### 14.3 消息格式

**文件名**：`YYYY-MM-DD-from-<source-project>-<topic-slug>.md`

- `<source-project>`：发送方项目短标识（如 `whoami`, `smss`）。
- `<topic-slug>`：主题的 kebab-case 概括。
- 示例：`2026-04-19-from-whoami-skill-improvement-proposals.md`。

**头部**（YAML frontmatter，必填）：

```yaml
---
from: <发送方项目名>
to: <接收方项目名>
date: YYYY-MM-DD
subject: <一行主题>
status: unread | read | actioned
in-reply-to: <原文件名，如为回复；否则 null>
priority: low | normal | high
---
```

**正文**：自由 Markdown。允许表格、清单、代码块。

### 14.4 消息生命周期与规则

```
发送方                                         接收方
  │                                              │
  ├─ 写入自己的 outbox/                           │
  ├─ 复制同一文件到接收方 inbox/                   │
  │  （发送方设置 status: unread）                │
  │                                              │
  │   ─── session 边界 ───                       │
  │                                              │
  │                      接收方的恢复链 ─────────┤
  │                      指向 inbox/             │
  │                      接收方读取消息 ─────────┤
  │                      → status: read          │
  │                      执行所需动作 ───────────┤
  │                      → status: actioned      │
  │                      （如需回复：            │
  │                       写一条新消息）         │
```

**规则**：

1. **收到的消息不可变**——永不修改 `inbox/` 内的文件，仅更新 `status` 字段。需回复就写新消息。
2. **status 转换**——发送方投递时设为 `unread`。接收方读后改为 `read`，执行完所需动作后改为 `actioned`。无需动作的消息直接 `actioned` 即可视为已确认关闭。
3. **归档**——`status: actioned` 超过 30 天的消息可迁入 `inbox/_archive/` 以保持活跃 inbox 可扫描。归档消息是永久记录，不得删除。
4. **发送方保留底稿**——发送方的 `outbox/` 是永久记录。若接收方丢失副本，以发送方 outbox 为权威。
5. **快照优于指针**——向其他项目通报数字、交付、状态（"我们模型精度达到 0.88"）时，把值直接写在消息正文里，而**不要**让对方指向你的内部文件。内部文件会随重构漂移，快照一旦投递不会说谎。

### 14.5 与四核心文档的集成

**CLAUDE.md** —— 添加一个铁律块（可直接复制粘贴，无需定制）：

```markdown
**项目间通讯走 inbox/outbox 文件协议**

- **机制**：本项目维护 `inbox/`（收信）和 `outbox/`（发信）。消息是 YAML frontmatter + Markdown 的 `.md` 文件，文件名格式 `YYYY-MM-DD-from-<source>-<topic>.md`。
- **为何强制（如已采用）**：跨项目信息仅在聊天中传递会在 session 结束时丢失。文件级消息跨越 session 边界保存，并通过本项目的恢复链被任何未来 agent 发现。
- **接收**：恢复链检查 `inbox/` 中 `status: unread` 的消息。读取 → 改为 `status: read`。执行动作后 → `status: actioned`。永不修改收到的消息（仅 status 字段），需回复则写新消息。
- **发送**：写入本项目的 `outbox/`（发送方永久底稿）并复制同一文件到接收方的 `inbox/`。在 CURRENT_STATUS 中记录该次通讯以便下次会话可追溯。
- **快照优于指针**：向其他项目通报数字/交付时，把值写在消息正文里，不要让对方引用你的内部文件。
- **完整规范**：`DOC_HARNESS_SPEC.md` §14（本项目根目录）。
```

**CURRENT_STATUS.md** —— 需要动作的入信加入车灯：

```markdown
### 立即行动
- [ ] 回复 <发送方> 的 <主题>（见 `inbox/YYYY-MM-DD-from-<source>-<topic>.md`）
```

发出的通讯随手记入车身：

```markdown
#### 向 <接收方> 发送交付物（YYYY-MM-DD）
- 消息：`outbox/YYYY-MM-DD-to-<target>-<topic>.md`
- 概要：<一行>
```

**FILE_INDEX.md** —— 两个目录单列类别：

```markdown
## 项目间通讯
- `inbox/` — 来自其他项目的消息
- `outbox/` — 发出的消息（底稿与已发副本）
```

**恢复链（CLAUDE.md）** —— 任务条件读加一条：

```markdown
### 任务条件读
- 如 `inbox/` 有任何 `status: unread` 的文件：优先处理
- ...
```

注意：这条在**任务条件读**层，不在必读。每次 session 开始把整个 inbox 读完会烧 context；只有 unread 消息要紧。inbox 为空或都是 read/actioned 时这条自然为 no-op。

### 14.6 给"首次遇到本机制"的 agent

如果你进入一个项目，在根目录看到 `inbox/` 和 `outbox/`，说明该项目使用本协议。请这样做：

1. 扫描 `inbox/`，找所有 frontmatter 中 `status: unread` 的文件。
2. 逐一处理：读取、执行所需动作（如有）、更新 `status` 字段（读后 `read`，完成后 `actioned`）。
3. 在修改本项目内部文件（可能被其他项目引用的模型、接口、数字）之前，考虑是否需要向依赖方 inbox 发一份快照。
4. 发送方法：写入本项目 `outbox/` 并复制同一文件到接收方 `inbox/`。在 CURRENT_STATUS 中记录。

你需要的一切都在本章。无需外部文件。

### 14.7 在已有项目中追溯启用

现有项目后续启用的步骤：
1. 在项目根创建 `inbox/` 和 `outbox/`。
2. 从 §14.5 复制铁律块到 CLAUDE.md 的铁律章节。
3. 把任务条件读条目加到恢复链（§14.5 中）。
4. 在 FILE_INDEX 加 `## 项目间通讯` 类别。
5. （可选）向已知依赖方项目发一条通知消息，告知本项目已可通过 inbox 联系。

| 信号 | 是否切换 |
|------|---------|

| 拿不准 | ❌ 不切换（避免过度维护），但在车身中用标题清晰分隔 |

- [ ] CURRENT_STATUS的"最后更新"是今天？

- [ ] 本session的工作步骤都记录在车身了？

## 附录E：设计选择（FAQ）

### 为什么 inbox/outbox 是可选机制，不是强制的？

因为很多项目是独立的，没有依赖方。强制每个项目都装载 inbox/outbox 会产生空目录和用不上的铁律，削弱对 skill 的信任。机制作为**标准可选特性**提供，附带完整自含规范（第十四章），需要的项目可一致采用，不需要的项目忽略即可。

### 为什么机制写在 doc-harness 内部，而不是独立 spec？

因为外部 spec 是**脆弱的**。项目内部工作的 agent 对外部文件没有可靠的入口；长期项目演化中，外部引用容易丢失、被误解、被遗忘。把完整机制嵌入 doc-harness spec 之后，每个部署了 `DOC_HARNESS_SPEC.md` 的项目都自带一份自含描述——项目自身成为"该协议在本项目里如何运作"的权威来源。

### 为什么 PARKING_LOT.md 和 PHILOSOPHY.md 是可选的？

不是每个项目都有值得记录的暂缓事项或跨项目原则。强制要求会产生空壳文件，削弱对四文档体系的信任。把它们作为模式提供，让每个项目在真的有内容时再采用。

### 为什么恢复链必读层保持在 2–3 条？

再大，"必读"就会变成"大概应该读"，agent 要么开始跳过条目，要么为保险起见读完整个项目。两层切分只有在基线真的精简时才起作用。只对*部分*工作相关的条目应放到任务条件读。

### 为什么用行数（而非阶段数）作为 WORKLOG 归档触发条件？

阶段密度在项目间差异巨大——从基础设施工作的每阶段 ~20 行，到密集研究工作的每阶段 ~200 行。行数反映读取此文件的实际成本，这才是我们关心的。
